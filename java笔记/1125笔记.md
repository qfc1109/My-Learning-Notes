### 5.锁

目前synchronized同步锁：把此代码，同步，加锁，本线程执行这一段代码，其它线程才可以进入此段代码

1 什么是锁：

类似于生活中的锁，一段代码加了锁，一但有一个线程在操作，

其它线程只能等待，再抢锁，再执行代码块

2 如果没有锁，比如异步，不用管，此段代码任何线程都是可以进入操作。

3 锁最大的好处：数据的安全

4 锁的原理：在硬件层面，cpu对锁提供 原子性操作  关中断 锁内存总线

原子性操作：让数据是最小单元，要么做完，要么不做，保证数据准确性。

关中断：禁止  启动 （上锁 让锁）

内存总结：通过信号量来判断 

5 锁缺点：效率低，经常要等待

6 场景：针对一些上锁的数据，才要上锁，取款。

7 无锁状态：异步  计算机默认状态

8 案例 卖票

先是无锁状态，导致多个窗口卖相同的票，对业务来说，是错的

### 6.悲观锁synchronized

synchronized: 不管是读还是写，本线程在操作时，其它线程都是进入不了。

代码层面：是把整个代码块同步了，业务有时是，其中一段须同步，另一段不用同步。不太灵活。

### 8.ReentrantLock又分公平锁和非公平锁

- - `ReentrantLock(boolean fair)`  根据给定的公平政策创建一个 `ReentrantLock`的实例。 

true: 公平锁，先到先得

false:非公平锁，不是按固定顺序，有时会插队(效率高，导致一种叫饿死)

true时，打印出时，thread-0  xx  thread-1  xx  thread-2 xx

false时，打印出时，全是乱的

多试几次，大概率是上面的

boolean 类型默认给的是false,所以默认是非公平锁

### 9.乐观锁：AtomicInteger

读数据时，不会上锁，当你操作数据时，上锁

AtomicInteger 原子类

操作数据时，只有一个线程在操作，可是当读时，所有的线程都是读到了最新的数据

### 10.读写锁:

ReentrantReadWriteLock

可以支持读锁和写锁，看场景

上读锁时，其它线程是可以过来读的，只是不能操作

上写锁时，其它线程连读都不可以过来，也叫互斥锁

- ```
  public class ReentrantReadWriteLock
  extends Object
  implements ReadWriteLock, Serializable
  ```

- ```
  public interface ReadWriteLock
  ```

  A`ReadWriteLock`维护一对关联的[`locks`](../../../../java/util/concurrent/locks/Lock.html)  ，一个用于只读操作，一个用于写入。 [`read  lock`](../../../../java/util/concurrent/locks/ReadWriteLock.html#readLock--)可以由多个阅读器线程同时进行， [`write  lock`](../../../../java/util/concurrent/locks/ReadWriteLock.html#writeLock--)是独家的。 

  

读写锁也可设置公平锁与非公平锁

- - `ReentrantReadWriteLock(boolean fair)`  创建一个新的 `ReentrantReadWriteLock`与给定的公平政策。 

true 公平锁

false 非公平锁

### 13.自旋锁

当一个线程尝试获得某个锁时，如果此锁已被其它线程占用，

一直在循环检测此锁是否被释放，不是进入挂起或阻塞，

缺点：自旋锁一直占着cpu,导致计算机性能下降

优点：当多个线程操作同一个变量，CAS操作，保证变量的值最新最准。

这是一种情况：避免此情况

AtomicReference

- ```
  public class AtomicReference<V>
  extends Object
  implements Serializable
  ```

  可以原子更新的对象引用

```
AtomicReference类提供了原子操作的类，也有cas
AtomicReference和AtomicInteger很像，AtomicInteger控制整数
AtomicReference可以控制很多对象
```

### 14.ABA的问题

多个线程，对某个值进行操作时，先把A改成B,

又把B改成A,当此时A 与A比时，计算机就认为此变量都未操作过。

产生的原因：

只是检测某个变量与期望值，判断相等，没有看到过程，

先把A改成B,

又把B改成A，只是错误的认为没有操作过。

问题：导致逻辑上的错误



```
compareAndSet（期望值，新值）
```

目前值

当目前值与期望值比较，如果相等就更新为新值
