



# Java面试宝典

### JVM，JRE，JDK 分别是什么

- **JVM**：虚拟出来的一台计算机，将字节码翻译成当前平台的机器码给平台执行
- **JRE**：运行JAVA程序的最小环境，JRE包含JVM加Java的基础类库
- **JDK**：开发JAVA程序的最小环境，JDK包含JRE加开发工具（编译javac.exe、运行java.exe等）

### java八种基本类型

- **整型**: 四种
  - byte   1个字节   8位    -2^7--2^7-1(-128~127)
  - short  2个字节   16位      ...
  - int    4个字节   32位      ...    常用的
  - long   8个字节   64位   -2^63--2^63-1
- **浮点型**: 两种
  - float    单精度   4个字节
  - double   双精度   8个字节      常用的
- **布尔类型**:   boolean     真true/假false
- **字符类型**:   char   可以存储单个字符  'A'  或者 ASCII码  或者 unicode 



### 自动类型转换和强制类型转换

- **自动类型转换** 小范围 转换大范围
  - 比如: int+double+String ==> String
  - 比如: int/double ==>double
  - 比如: double=int        long=int
- **强制类型转换**: 大范围 转换小范围
  - 比如: int i=(int)15.5;  



### && 和 & 区别

- **&&** 具有短路功能 ,如果条件1 不满足  条件2不会执行   所以执行效率稍微高一些
- **&**  没有短路功能,如果条件1 是否满足 条件2都会执行



### break 和 continue 和 return 区别

- **break**: 可以跳出当前循环  也可以用在switch 中跳出当前switch
- **continue**:  一般用于再循环中 用于跳出本次循环 执行下一次
- **return** :  表示结束当前方法



### java为什么可以跨平台

- Java程序的代码文件需要先编译成字节码文件，字节码文件再通过不同硬件平台上安装的对应版本的JVM虚拟机翻译成当前平台的机器码，交给当前平台执行, 因此，不同的系统平台只需要安装对应系统版本的JVM虚拟机，就可以运行相同的Java程序，就实现了跨平台运行



### 排序算法

- **冒泡排序**: 两两比较排大小

  ```java
  //外层控制比较轮数
  for(int i=0;i<数组.length-1;i++){
  	//内层控制每轮比较次数
  	for(int j=0;j<数组.length-1-i;j++){
  		if(nums[j]>nums[j+1]){
  			//两值互换
  			nums[j]=nums[j]^nums[j+1];
  			nums[j+1]=nums[j]^nums[j+1];
  			nums[j]=nums[j]^nums[j+1];
  		}
  	}
  }
  ```

- **选择排序**: 第一个和其他元素比较  求出最小值  然后 第二个和其他元素比较 求出第二小的值

  ```java
  for(int i=0;i<数组.length-1;i++){
  	int index=i; //记录下标的
  	for(int j=i+1;j<数组.length;j++){
  	    //找到比我index小的元素
  		if(nums[index]>nums[j]){
  			index=j; //记录下标
  		}	
  	}
  	//经过内层循环 知道 index是目前最小的
  	if(i!=index){
          nums[i]=nums[i]^nums[index];
          nums[index]=nums[i]^nums[index];
          nums[i]=nums[i]^nums[index];
  	}
  }
  ```

- **插入排序**: 将排序的数组 分成两部分:有序部分和待插入部分,将待插入的部分 和前面有序部分 进行比较 找出插入的位置

  ```java
  for(int i=0;i<nums.length-1;i++){
      //j=i+1 原因从第二个元素开始
      //j>0   从后面往前比较 
  	for(int j=i+1;j>0;j--){
  		if(nums[j]<nums[j-1]){
  			int temp=nums[j];
  			nums[j]=nums[j-1];
  			nums[j-1]=temp;
  		}
  	}
  }
  ```

- **二分查找**:  前提是在一个有序数组的基础上 进行折半查找  

  - 定义两个变量  min=0  max=最大下标,求出中间值  (min+max)/2   通过中间值和判断元素比较
  - 如果相等 表示 找到了 则应该结束
  - 如果大于 应该继续查找中间值右边   更新min=中间值+1,更新后继续求新的中间值比较
  - 如果小于 应该继续查找中间值的左边  更新max=中间值-1,更新后继续求新的中间值比较

  ```java
  public static int test4(int[] nums){
          System.out.println("测试二分查找查找");
          int number=1000;
          //初始化 最小 和最大下标
          int min=0;
          int max=nums.length-1;
          while(min<=max) {
              //求出中间值下标
              int mid = (min + max) / 2;
              if (number > nums[mid]) {//查找的元素大于中间值
                  min = mid + 1; //最小下标更新中间值+1
              } else if (number < nums[mid]) {
                  max = mid - 1; //最大下标更新中间值-1
              } else {
                  return mid;
              }
          }
          return -1;
      }
  ```

- **快速排序**: 原理是给基准数找到正确的索引值，把比基准数大的元素放在右边,把基准数小的元素放在左边，再通过递归对前半部分和后半部分再进行排序,这样数组就有顺序了

  - 假设设置一个数是基准数 通过temp=nums[0];然后分别从两端扫描数组
  - 设置两个指针begin表示开始的指针  end结束的指针
  - 先从后开始扫描数组,如果扫描的值大于基准数  让end-1 ,再继续比较  如果扫描的值小于基准数  将end位置的值,赋值给begin位置的值
  - 再从前开始扫描数组，如果扫描的值小于基准数 begin+1 ,在继续扫描
  - 如果扫描的值大于基准数  把begin位置的值赋值给end位置

- 如果end和begin相等了 该位置就是基准数的位置,最后通过递归给基准数前半部分和后半部分 在按照这个规则排序 整个数组就有顺序了



### 排序复杂度

- o(1): 表示最低复杂度 常量值 消耗时间和内存大小和数据量没有关系  无论多大的数据 只要计算一次(比如:哈希算法)
- o(n): 数据量越大 耗时会增加n倍(比如:遍历循环)
- o(n^2): 对n个数排序 需要扫描n*n次(比如:冒泡排序)
- o(logn): 当数据量增大n倍 消耗会增大logn倍(比如:数据是256个 销毁会增大8倍  2的8次方=256   二分查找就是logn)
- o(nlogn): 就是n*logn 如果数据量增加256倍 消耗会增大
      256*8=2048 (比如: 快速排序)

| 排序算法 | 平均时间复杂度 | 空间复杂度 | 稳定性 |
| :------: | :------------: | :--------: | ------ |
| 冒泡排序 |     o(n^2)     |    o(1)    | 稳定的 |
| 选择排序 |     o(n^2)     |    o(1)    | 不稳定 |
| 插入排序 |     o(n^2)     |    o(1)    | 稳定的 |
| 快速排序 |    o(nlogn)    |  o(nlogn)  | 不稳定 |
| 二分查找 |    o(logn)     |            |        |

### 成员变量和局部变量区别

- 作用域不同: 成员变量作用在全类中,局部变量作用在定义范围之内 比如:if里面定义作用在if
- 初始值不同: 成员变量有默认值 不赋值也可以使用,局部变量没有默认值 所以使用前必须先赋值
- 优先级不同: 如果局部变量和成员变量同时存在  ,局部变量优先于成员变量(就近原则)



### this和super关键字

- this: 表示当前类的对象  
  - this.属性名      调用类中的成员变量
  - this.方法名()     调用类中的普通方法
  - this()    调用本类无参构造方法  只能写在构造方法里面,而且必须放在第一行
  - this(参数)  调用本类有参构造方法  只能写在构造方法里面,而且必须放在第一行
- super: 表示当前类的父类对象 它的 用法和this 一样只不过是针对于父类      



### 方法重载overload和重写override区别

- 重载overload: 一般情况下在一个类中,要求方法名必须是一致的,参数列表是不一致的(类型  个数  顺序)  跟返回值无关
- 重写override:  发生在具有父子类的关系中,要求  方法名 和参数 都一致  访问修饰符相同或者不能严于父类   返回值类型相同 或者 可以是其子类类型   不能抛出比父类更多的异常



### 面向对象基本特征

- 封装:  将类中的属性私有化(private),可以让其他类不能直接使用我的属性,对外提供两个公开的(public)方法 分别用于设置属性setXX()和 获取属性的getXX() 就可以通过自己提供的方法 
  去设置 获取和设置属性规则  就可以防止数据非法的 提高了安全性

- 继承:  将很多类共同特征(属性和方法)提取出来通过一个类完成, 这样其他类就可以继承这个类 就可以使用这个类共同特征   可以提高代码的重用 

- 多态:  同一个引用类型,使用不同的实例执行不同的操作  一般要多态需要先 继承   方法需要重写   然后 父类引用指向子类对象 

  - 比如: 动物类的子类有狗类和猫类 , 都具有吃饭的方法    使用时就可以通过

    声明动物类 指向 狗类对象 或者 猫类对象   就可以调用 狗 或者 和猫不同的吃饭方式

    

### final 关键字用法 

final类似于static都是表示修饰符,只是修饰的含义不同,只表示最终的  可以修饰属性 方法  类

- final修饰的类 表示最终类 不能被其他类继承
- final修饰的方法  表示最终方法 不能被重写
- final修饰的属性 表示最终属性  也就是常量   表示值一旦赋值不能够被改变  所以使用前必须先赋值



### 接口和抽象类 区别

- 抽象类可以有普通属性 接口都是常量
- 抽象类可以有非抽象的方法 接口全是抽象方法
- 抽象类可以有构造  接口没有构造方法
- 接口和接口是多继承 类和类是单继承 但是类可以多实现接口
- jdk1.8以后支持默认方法和静态方法

接口和抽象类 应用场景

- 抽象类: 
  - 类之间如果是包含的关系  (比如 狗  属于 动物 类) 并且需要共享代码，用**抽象类**
  - 若仅需单一继承体系，用**抽象类**
  - 若需要实例变量、构造方法初始化状态，用**抽象类**
- 接口: 
  - 若仅需统一行为(比如: 都可以吃饭   睡觉) 使用接口
  - 一个类需要具备多种不相关的能力（如 “游泳 + 飞行”），用**接口**（多实现）
  - 若仅需定义行为规范，用**接口**



### 成员变量(实例变量)和静态变量区别

- 成员变量: 属于 对象的   随着对象的创建 而创建  存储再堆内存  一般通过对象名调用  
- 静态变量: 属于 类的   随着类加载 才会创建    存储再方法区   可以通过对象名调用也可以通过类名调用  



### 堆(heap)和栈(stack)的区别

- 堆: 一般用于存放new关键字创建的对象实例，对象实例不会随方法的结束消失，而是由垃圾回收机制处理
- 栈: 当程序进入一个方法时，会为这个方法单独分配一块私属存储空间，叫做栈帧，用于存储这个方法内部的局部变量等，当这个方法结束时，对应栈帧释放，栈帧中变量随之释放



### JVM内存模型

- 线程独享
  - 程序计数器:  每个线程都会有独立的程序计数器,记录当前线程执行字节码指令的位置(行号) 目的是为了线程恢复执行的时候 可以再原来行下面继续执行  
  - 虚拟机栈: 每个线程都会有一个独立的栈,每个栈 会包含很多个栈帧，栈帧是用来控制每个方法的执行过程    由于方法和方法调用满足先进后出 所以栈的执行规则 也是先进后出
    - 栈帧主要包含: 

      - 局部变量表：用来保存再方法中定义局部变量
      - 操作数栈:   用来实现方法内部做的运算(+ -* /)
      - 动态链接:   指的是程序执行过程中 字节码和方法链接的调用过程   方法的调用和绑定 
      - 返回地址:  不是方法返回值  是表示方法调用之后 执行完还要返回到你调用的位置 
  - 本地方法栈：类似于虚拟机栈 区别在于 虚拟机栈 是控制我们定义的方法执行过程  而本地方法栈 是控制native方法(本地方法)

    - native方法: 就是c++ 或者其他非java语言实现出来的方法  java只负责调用 不用考虑谁实现的
- 线程共享:
  - 堆内存: 堆是jvm内存最大一块区域  用于存储创建出来的实例(对象)  是被所有线程共享的  （比如: new User()  new 集合  new int[5]）,堆的大小可以通过jvm启动参数  比如(-Xms(初始堆大小)   -Xmx(最大堆大小)) 来控制
  - 方法区: 主要存储类相关的资源  比如类的信息  类中属性 类中方法   常量池(保存常量)  静态变量   类编程后的代码也会保存 

### GC垃圾回收

GC就是java垃圾回收的缩写    目的就是再jvm中管理和回收堆内存的对象   一般用于 回收 java中没有任何人引用的对象  GC再回收对象时  会通过一定的策略和方法   

主要包含两种 （`1.GC标记算法` ,`2.GC回收算法`）

### GC标记算法有哪些?

- `引用计数法`: 它 是一种最简单标记算法 ,目的是用于追踪这个对象是否可用  每个堆内存存储的对象 都有一个引用计数器 当一个变量指向了它  计数器+1  如果少了一个变量指向它  计数器-1   当计数器为0   那么GC就可以把它标记成可以回收的对象

  - 缺点: 无法处理循环引用的问题  如果多个对象相互引用  计数器永远不会为0   除非有外力介入  否则不会回收的  很容易出现内存溢出 

- `可达性分析法`: 首先它是jvm用的比较广泛的标记算法  用于判断对象是否是可达的   原理是:  首先从根(GC root)开始 作为整个对象访问链的入口 , 从根开始向下搜索(深度优先搜索)  就会形成上面哪种 引用链   这样再引用链的对象是可以存活的    反之不在引用链上的对象 就是不可达对象(可以被回收)

- `三色标记法`:  jvm是最常用的标记算法   会给堆内存的对象设置三种不同的颜色  通过不同的颜色 来区分是否可达

  - 白色:  表示垃圾回收器 没有被访问过的对象  这些对象可能是垃圾

  - 灰色:  表示垃圾回收器  已经访问过 它是可达的  但是和它关联的对象还没有扫描完毕 

  - 黑色:  表示垃圾回收器 已经访问过 它是可达的  同时和关联的对象也全部访问过  从灰色变成黑色

  - 执行原理

    - 首先,所有对象默认都是白色

    - 从GC root根开始  扫描下一级  如果扫描到了 立马变成灰色

    - 然后继续采用 深搜和广搜   优先扫描  灰色相关联的对象   如果扫描完毕 

      从灰色变成黑色

    - 最终扫描完毕   剩余白色对象  就是不可达对象 就可以被垃圾回收

    

### GC回收算法有哪些?

- `标记清除法`:  是gc中回收算法中最基础的  主要分两个阶段

  - 标记阶段: 垃圾回收器 从根开始 进行搜索 找出哪些对象是可达的(可达性分析法 )  找到之后 给这些对象打上一个标记
  - 回收阶段: 除了哪些打上标记的对象   进行统一回收
    - 缺点: 一旦采用这种回收算法  会导致内存碎片特别多  这样会导致 后期需要分配一些大连续空间内存(数组)  很可能出现内存不足

- `复制算法(拷贝算法)`:  一般适用于新生代对象, 会将内存划分两个区域  一个是可用内存和保留内存 ，首先对象的创建再可用内存保存   也需要经过标记阶段 找出哪些对象可达     然后将可达对象移动到保留内存中紧密排在一起   最后针对可用内存整体回收 然后变成新的保留内存   而之前的保留内存变成可用内存

  - 缺点:  内存一分二  内存利用率减半, 如果针对于生命周期比较长的对象，频繁复制 会浪费性能

- `标记整理法(标记压缩法)`: 能够完美解决 标记清除法出现内存碎片  和 复制算法出现的内存减半的问题  主要分两个阶段

  - 标记阶段：类似于标记清除法的标记阶段  从gc root根开始找到哪些是存活的  对存活的对象进行标记
  - 压缩阶段(整理阶段):  将标记存活的对象 进行压缩 移动内存的一端 使对象紧密排在一起   压缩完毕后  其他内存区域整体回收
    - 缺点: 由于每次都需要进行压缩 来不断移动存活对象 ，如果存活对象很多 移动性能也会消耗很多 比较浪费性能

- `分代算法`: GC会将创建的对象 根据回收次数 或者对象大小 来划分成不同的区域 主要有两个 1.新生代   2.老年代

  - 新生代: 新创建的对象一般存储新生代的   如果 创建对象比较大 可能会直接晋升老年代   新生代由于刚刚创建 存活周期不是很长   会需要频繁回收就适合采用复制算法  新生代主要三个区域（Eden区 8 From区  1 To区 1）

    - Eden区:   用于存活刚刚新建的对象,如果创建的对象比较大 这里不存储了 直接保存到老年代  如果Eden区 内存不足 触发GC进行新生代垃圾回收
    - From区: 一般作为上一次GC存活者  下一次GC的扫描者
    - To区: 一般是表示保留内存 包含一些已经被清理的对象

  - 老年代: 在新生代将进行一定次数(15)的GC依然存活的对象   或者对象比较大 都会存储到老年代    所以老年代的对象都是相对比较稳定   存活周期都比较长 比较适合采用 标记清除法 和 标记压缩法

    

  - from区和to区如何相互转换：

    - 首先垃圾回收器 标记出 Eden区 和from 区 存活对象 然后把它存活的对象复制到To区

    - GC回收后, 原来的from区 变成空了  因为存活的对象已经 To区保留好了

    - 原来To区 保存了Eden区 和from区存活的对象   就会把To区转换From区

      之前From区变成新的To区

    - 下一次GC 所有新的对象也会创建在Eden区  然后GC时继续标记 Eden和新的From区存活对象   标记好了 复制给新的To区

    
  
  - 新生代和老年代转换
  
    - 对象保存在新生代 经历了一定次数的GC   每经历过一次GC年龄就+1  当年龄达到15岁 依然存活 直接晋升老年代
    - 如果创建的对象比较大  直接跳过新生代 进入老年代
    - 新生代中有from区和to区  对象总量超过一半时   因为他们的年龄有一个阈值   也会直接晋升老年代    避免 from和to区 内存不足  出现频繁复制的问题



### 设计模式

 保证一个类只能创建一个对象(实例), 原因很多需求其实并不需要多个实例 (比如：servlet默认单例模式   sqlSession工厂只需要一个就可以创建无数个sqlSession)   如果没有设计单例模式  每用一次 多个新的对象  就会浪费不必要的资源 ...

- 静态内部类: 是通过类加载方式的实现的, 因为类加载的过程本身   就是线程安全的(类只需要加载一次) 所以它就无需添加synchronized   同时类加载 也是具有懒加载的功能 什么时候需要这个类 才去加载它

### 类加载器分类

- **`BootstrapClassLoader`(启动类加载器)**：最顶层的加载类，由 C++实现，通常表示为 null，并且没有父级，主要用来加载 JDK 内部的核心类库（ `%JAVA_HOME%/lib`目录下的 `rt.jar`、`resources.jar`、`charsets.jar`等 jar 包和类）以及被 `-Xbootclasspath`参数指定的路径下的所有类
- **`ExtensionClassLoader`(扩展类加载器)**：主要负责加载 `%JRE_HOME%/lib/ext` 目录下的 jar 包和类以及被 `java.ext.dirs` 系统变量所指定的路径下的所有类
- **`AppClassLoader`(应用程序类加载器)**:  面向我们用户的加载器，负责加载当前应用 classpath 下的所有 jar 包和类

这些类加载器之间采用**双亲委派模型**：当一个类加载器收到加载请求时，会先将请求委派给父类加载器处理，只有当父类加载器无法加载时，才由自己尝试加载



### 类加载器的特点

- **双亲委托**：当一个自定义类加载器需要加载一个类，比如 java.lang.String，它很懒，不会一上来就直接试图加载它，而是先委托自己的父加载器去加载，父加载器如果发现自己还有父加载器，会一直往前找，这样只要上级加载器，比如启动类加载器已经加载了某个类比如 java.lang.String，所有的子加载器都不需要自己加载了。如果几个类加载器都没有加载到指定名称的类，那么会抛出 ClassNotFountException 异常
- 负责依赖：如果一个加载器在加载某个类的时候，发现这个类依赖于另外几个类或接口，也会去尝试加载这些依赖项
- 缓存加载：为了提升加载效率，消除重复加载，一旦某个类被一个类加载器加载，那么它会缓存这个加载结果，不会重复加载



### 双亲委派

- 如果一个类加载器在接到加载类的请求时，它首先不会自己尝试去加载这个类，而是把这个请求任务委托给父类加载器去完成，依次递归，如果父类加载器可以完成类加载任务，就成功返回。只有父类加载器无法完成此加载任务时，才自己去加载

  - 本质：规定了类加载的顺序。引导类加载器先加载，若加载不到，由扩展类加载器加载，若还加载不到，才会由系统类加载器或自定义的类加载器进行加载

### 迭代器底层原理

Java迭代器通过‌**游标跟踪**‌和‌**Fail-Fast机制**‌，提供了一种安全、统一的集合遍历方式。其核心思想是‌**封装集合的遍历逻辑**‌，使得遍历过程与集合实现解耦，同时防止并发修改导致的数据不一致，迭代器是Collection接口的方法(list和set集合才可以直接使用迭代器遍历  map不是 它不能直接使用...)

- 底层原理: 
  - 游标跟踪: 迭代器通过一个‌**游标（cursor）**‌跟踪当前遍历的位置 , 
  - Fail-Fast机制:迭代器内部还可以 防止在迭代过程中其他线程或操作修改集合结构
    - 集合内部维护一个 `modCount`（修改计数器），每次集合结构变化（如添加、删除）时递增,代器创建时记录当前的 `modCount` 为 `expectedModCount`每次调用 `next()` 或 `remove()` 时检查两者是否一致，不一致则抛出 `ConcurrentModificationException`

- Collection: 是集合父接口  子接口有List 和 Set  提供很多集合正常操作的方法 比如: add()  remove() size()...
- Collections: 是集合工具类   类似于Arrays  都属于util包的资源   提供了很多集合写好功能的静态方法  比如:  sort() 排序shuffle()  随机打乱  binarySearch(集合,元素) 二分查找  



### HashSet或者HashMap底层原理

- 数组: 采用连续空间存储数据 通过下标查找元素  查询时间复杂度O(1)  ,遍历数组  由于要逐一遍历 复杂度O(n)  不适合增删
- 链表: 采用不连续空间   通过节点 指向下一个节点得，适合增删  不适合查询
  - 单向链表:  包含数据 和 next指向下个节点地址
  - 双向链表:  包含数据 和 prev指向上个节点地址和next指向下个节点地址 
- 哈希表: 也叫散列表 ,就是集合了数组查询快 和 链表增删快得特点  ，在哈希表中 插入 删除 做查询(不考虑哈希冲突) 时间复杂度都是O(1) , jdk1.8 后 哈希表又引入红黑树
- 红黑树:  红黑树是一个自我平衡有序二叉树，普通平衡二叉树是通过高度差不大于1来控制平衡得  , 红黑树是每个节点设置不同红色和黑色 通过颜色达到自我自我平衡 ,特点如下:
  - 根节点 必须是黑色得
  - 每个存储null叶子节点 也是黑色得
  - 如果有一个节点是红色得 那么它得子节点只能是黑色 因为不允许出现两个连续得红色
  - 保证从根节点开始 到达任何一个叶子节点  黑色数量是一样得  从而达到颜色平衡

>是基于哈希表实现的 数组+链表+jdk1.8红黑树构成的,默认长度16 第一次进行put()存值时 才初始化16 有一个加载因子0.75 会通过长度*加载因子计算扩容阈值,当集合超过这个阈值就会扩容2倍    ,它存储元素原理(put()) ,先通过==判断 地址是否一样 再调用hashcode()计算哈希值  , 通过哈希值做位运算求出数组下标,  最后调用equals() 判断key的内容就那个是否相同    如果值也相同  说明是重复的元素  进行覆盖   如果值不同  就属于hash冲突  就需要通过next指向下一个节点 形成了链表  ,最后链表长度超过8  并且数组元素个数超过64  如果再继续存储会影响性能  就会把链表升级为红黑树  ，  如果红黑树高度小于6   就会树降级成链表



### HashMap 和Hashtable区别

- HashMap:   基于哈希表实现   允许key和value是null

  默认长度16  加载因子0.75  扩容2倍

  线程不安全得   效率高

- Hashtable:  基于哈希表实现   不允许key和value为null

  默认长度11 加载因子0.75  扩容2倍+1

  线程安全得   效率低 

### ConcurrentHashMap底层原理

属于线程安全的HashMap   底层jdk1.7采用分段锁来实现   会将数据分成很多个数据段   如果针对当前数据段加锁   对于其他数据段的使用没有影响  
底层jdk1.8采用node节点来加锁 锁的颗粒度会比分段锁更加细致  如果针对于当前节点加锁  对于其他节点进行操作没有影响





### 线程和进程的区别

- 进程:  代表一个程序的执行过程,包含了程序所持有资源和线程的集合   一个情况下一个程序只有一个进程  每个进程也会有自己独立空间   所以先运行期间 可以互不影响
- 线程:  线程属于进程的一部分   它是用于程序执行某项特定的任务   所以一个进程 会包含无数个线程
- 多线程并发:  表示多个线程交替抢占CPU资源  并不是同时执行    由于切换CPU时间非常短   正常无法发现 他们交替执行  所以用户看起来以为是同时进行     为了提高用户体验



###  什么是守护线程

- 用于服务用户线程的,如果用户线程结束了  守护线程也会自动结束  为了给用户线程提供很多的帮助  类似于用户线程保姆   比如: gc垃圾回收 就是守护线程
  - 实现方式: 类似于编写用户线程,需要设置t1.setDaemon(true)  , 前提不要写在start() 前面 否则守护线程设置失效



### 创建线程的方式

- 继承Thread类 ,重写run()
- 实现Runnable接口，实现run()
- 实现Callable接口, 实现call()
- 利用线程池来创建:  一个容器(空间 内存区域 )  先创建很多你需要得对象(线程  连接...)   目的就是如果你需要使用  可以直接再池中获取  使用完毕之后还可以自动还会池中  为了保证池中对象得数量  如果并发量很高 可以创建更多得对象  供你使用   如果并发量很少  也会回收一些无用得对象..   这样就节省了  我们自己创建对象得时间      它是以空间换时间得方式来提高效率
  - 自带线程池
    - 可缓存线程池:  空间可以无限大  当并发量比较高 为了满足需求就会创建更多的线程  当并发量比较少  为了节省空间 也会回收一些长时间不用的线程
    - 可定长线程池: 定义长度固定的线程池，所以最高并发数量 是固定的 如果超过了并发数量  则会等待其他线程使用完
    - 可周期定长线程池: 具备上面定长的功能   同时还支持 周期操作(可以每隔一段时间都去执行任务)  和延迟操作(首次延迟的时间)
    - 单线程池: 线程池只会创建一个线程  支持单个线程做并发任务的(比如:排队购票 只有一个购票窗口 )
  - 自定义线程池:  前面四种线程池 基本可以满足 大部分主流的需求 肯定有满足不了的  这时就可以通过ThreadPoolExecutor来创建线程池 传入七大核心参数 来满足用户的需要



### Runnable接口和Callable接口区别

- Callable实现的方法是call() ， Runnable实现的是run()
- call()可以提供返回值  run()没有返回值
- call()可以抛出异常  run() 只能捕获异常 不能抛出
- Callable需要依赖FutureTask对象封装才可以使用，他的底层就是Runnable接口类型



### 线程方法start()和run()区别? 启动线程使用start()还是run() 为什么?

- start(): 用于启动线程 让线程进入就绪状态, 启动线程必须使用start()   可以让线程和其他线程竞争CPU
- run(): 只是表示线程执行逻辑  启动线程不能使用run() 只相当于对象调用一个方法而已



### sleep()和wait()区别?

- sleep: 属于Thread类静态方法  让线程进入休眠(阻塞状态) 放弃当前CPU执行权,休眠过后会进入就绪状态(可以竞争CPU)  并且不会释放对象锁
- wait: 属于Object类的方法 让线程进入等待(阻塞状态)  必须需要其他线程唤醒才可以执行  否则会一致等待   会释放对象锁



### 线程状态(线程生命周期)

- 新建状态: 当new  Thread() 创建了新线程 处于新建状态
- 就绪状态: 线程调用start() 就会进入就绪状态  它就可以其他线程竞争CPU
- 运行状态: 当线程抢到了CPU之后 就可以自动运行run() ,就会处于运行状态
- 阻塞状态: 线程再运行过程中  由于一些特定原因 导致了线程会暂时停止执行 ，阻塞原因:
  - sleep  不睡醒不执行
  - wait  需要别人唤醒
  - join  需要线程执行结束后才执行当前线程
  - Scanner 需要等待用户输入输出
  - futureTask.get()  需要等待call执行结束了 有返回值才走
  - GC:  等待堆内存不够了 触发GC执行
  - 线程同步锁: 需要等待其他线程用完了释放锁 才可以竞争
  - socket.accept() 等待客户端连接
  - ....
- 死亡状态: 
  - 正常死亡:  run()正常运行结束了 
  - 异常死亡: run() 运行时出现了异常 没有捕获解决 导致程序中断





### 什么是线程安全 ? 用过线程安全的类？

- 线程安全:  无论是单线程 还是多线程 执行 最终的结果都会保持一致  不会多线程并发修改的问题  这种情况就是线程安全  ,java底层大部分实现安全 都是通过Synchronized添加同步锁 ,因为添加它之后  想执行的话 需要先获取锁 ，如果获取到了  就可以执行  其他线程就需要等待 该线程 使用完释放了锁 才可以继续竞争锁  这样就会导致多线程并发执行  转换成让线程一个一个执行 好处是可以解决并发问题  坏处执行效率很低
- 实现了线程安全的类:  StringBuffer    Hashtable   ConcurrentHashMap



### 线程同步锁

- `排他锁/互斥锁`：用于控制共享资源访问的一种锁，它允许同一时间点只有一个线程可以访问(读写)共享资源 其他线程必须等待释放了锁之后 再竞争锁资源再访问
- `读写锁`：  读写锁是ReentrantReadWriteLock类,包含了读锁和写锁, 其中读锁允许多个线程同时访问 因为读取操作本身就是线程安全的 但是写锁是互斥的   总结: **读读不互斥  读写互斥  写写互斥**
- `公平锁/非公平锁`:  公平锁就是让线程进行排队等待的锁机制,先来的线程 先执行 后来的线程后执行  不会出现插队的情况   保证线程按照竞争锁的顺序 来获取锁  这样每个线程 都有机会执行 ReentrantLock   ReentrantReadWriteLock 都可以设置公平锁 
- `乐观锁和悲观锁`:  
  - 乐观锁:   通过数据库表中添加一个版本号 或者 时间戳 字段, 每次修改之前  会先查询版本号 或者时间戳    当真正修改时 会拿前面查询出的版本号 或者时间戳 作为条件 进行修改  如果两次版本号 不一致 说明被其他人修改过  就不能修改 如果两次版本号 是一样 说明没有其他人修改过  就可以修改  最后修改成功后 更新版本(版本号+1)   `类似于 迭代器 底层modCount控制并发修改的原理`
  - 悲观锁:  本身是数据库实现的机制   每次操作数据时  会把树锁起来   那么其他人就无法操作这些数据  最后数据库事务提交的时候   才会把锁释放   其他人才可以操作这些数据   `类似于互斥锁`



### 什么是死锁

就是多个线程 相互持有对方的锁资源  都需要等待对方来释放锁 才可以获取  由于释放不了    这样就会导致了线程一直处于等待获取锁的阶段   时间长了 就属于死锁了



### 死锁四个必要条件

- 互斥条件: 一个资源一旦分配给了某个线程 ,其他线程就不能访问  直到该线程释放锁为止(我再用别人不能用 用完了别人才可以用)

- 请求与保持条件:  一个线程对自己持有的资源保持不放   自己不满足 还想获取其他资源 ( ..... )

- 不可剥夺条件: 对已经分配的资源 其他线程不可以强制剥夺  只能本线程自己主动释放

- 循环等待条件: 有多个线程 相互持有对方的资源   就会构成一个循环链 

  

### 如何避免死锁

- 破坏互斥:   可以对于一些特定资源  可以允许多个线程同时访问 
- 破坏请求与保持:  线程再申请资源的时候  可以一次性申请所有资源  如果无法满足  可以先主动释放自己占有的资源   再申请新资源
- 破坏不可剥夺:  当一个线程 占有资源   如果想申请新的资源被拒绝   可以借助于外力强制性剥夺他  以满足其他线程的需求
- 破坏循环等待:  可以对申请的资源进行编号  线程再申请资源时可以按照 升序 进行申请   从而避免了循环



### synchronized和lock锁区别

- synchronized是Java自带的关键字,Lock锁是jdk1.5支持的接口
- synchronized使用比较简单 可以自动加锁和释放锁,Lock锁必须手动加锁(lock()) 和 手动释放锁(unlock())
- synchronized是属于非公平锁,Lock锁提供了轮询机制可以实现公平锁
- synchronized尝试获取锁时 如果获取不到 它会一直等待   Lock锁有一个方法tryLock() 可以控制等待时间 超时则不等待了
- synchronized唤醒线程时 只能随机唤醒一个 或者全部  Lock锁是可以精确唤醒  也可以分组唤醒 
- 性能方面:  要考虑到竞争是否激烈 , 如果竞争很激烈  那么Lock锁的性能会高于Synchronized(原因synchronized会根据竞争激烈程度将锁进行升级从偏向锁-->轻量级锁--->重量级锁, 而重量级锁的它的性能会低于Lock锁) 如果竞争不激烈 synchronize d就不需要锁升级  那么偏向锁的性能会高于Lock锁



### synchronized的锁升级完整版

`synchronized` 的锁升级是 JVM 为了平衡线程安全与性能而设计的优化机制，其核心逻辑是**根据竞争激烈程度，从低开销的锁形式逐步升级为高开销但更安全的形式**，且锁只能升级不能降级。整个过程从 “无锁” 开始，依次经历**偏向锁 → 轻量级锁 → 重量级锁**三个阶段

- **无锁状态(初始状态)** :就是对象刚被创建，尚未被任何线程锁定

- **偏向锁(单线程竞争)** : 当只有一个线程多次获取锁时，JVM 会启用偏向锁，减少无竞争场景的同步开销

  - 升级规则: 线程第一次获取锁时，通过 **CAS 操作**将对象头 Mark Word 中的 “偏向线程 ID” 设置为当前线程 ID，并将锁标志位改为 “偏向锁”（标志位 `01`，偏向标志 `1`）
  - 后续获取锁:该线程再次获取锁时，无需 CAS 操作，仅需判断对象头的 “偏向线程 ID” 是否为自身
    - 是：直接获取锁，几乎无开销
    - 否：触发偏向锁撤销（升级为轻量级锁）

- **轻量级锁(轻度竞争)** ： 当有多个线程交替获取锁（非同时竞争）时，偏向锁撤销后升级为轻量级锁，通过自旋避免线程阻塞

  - 升级规则: 偏向锁撤销后，竞争线程会在自己的栈帧中创建**锁记录（Lock Record）**，存储对象头的拷贝

  - 线程通过 **CAS 操作**将对象头的 Mark Word 替换为指向自己锁记录的指针

    - CAS 成功：当前线程获取轻量级锁（锁标志位 `00`）

    - CAS 失败：说明有其他线程竞争，进入自旋重试

      

  - 自旋机制: 竞争线程不会立即阻塞，而是通过循环（自旋）尝试获取锁（默认自旋 10 次，可通过 `-XX:PreBlockSpin` 调整）。自旋的意义是：线程阻塞 / 唤醒的开销很大，短时间自旋可能让线程快速拿到锁（适合竞争时间短的场景）

  - 升级为重量级锁: 

    - 若自旋次数超过阈值，或自旋线程数过多（如超过 CPU 核心数的一半），轻量级锁会升级为**重量级锁**。
    - 此时，未获取锁的线程会被阻塞（进入内核态），等待持有锁的线程释放后唤醒。

    

- **重量级锁(激烈竞争)**: 当多个线程同时争抢锁（激烈竞争）时，轻量级锁升级为重量级锁，依赖操作系统的互斥量（Mutex）实现同步

  - 原理:
    - 重量级锁的对象头 Mark Word 指向操作系统层面的互斥量（内核对象），锁标志位为 `10`
    - 未获取锁的线程会被放入锁的**等待队列**（阻塞状态），不再消耗 CPU 资源
    - 持有锁的线程释放锁后，会唤醒等待队列中的一个线程（或多个，取决于唤醒策略）



### synchronized的锁升级简化版

- 锁升级的基本流程:

  - **初始无锁**：对象创建后处于无锁状态。
  - **首次获取**：线程 A 首次获取锁，升级为偏向锁（对象头记录线程 A ID）。
  - **再次获取**：线程 A 再次获取，直接通过线程 ID 校验，无需额外操作。
  - **其他线程竞争**：
    - 线程 B 尝试获取锁，触发偏向锁撤销。
    - 若线程 A 已释放，升级为轻量级锁，线程 B 通过 CAS 竞争。
    - 若线程 A 未释放，线程 B 自旋重试。
  - **竞争加剧**：自旋失败，轻量级锁升级为重量级锁，线程 B 进入阻塞队列

  



### volatile关键字特性

- volatile不能保证原子性

  - volatile是用于修饰属性得 用于将工作内存得值刷新到内存  如果对于变量得操作 是属于复合操作(比如: i++ 等价于 i=i+1(先读取 再计算 最后赋值)) 它无法保证 这些操作同时执行  所以它保证不了原子性   也保证不了线程安全  

- volatile可以防止指令重排序

  - 程序为了提高优化性能 再多线程情况下可能会出现乱序执行代码得情况  这样就会造成 编写代码得顺序 并不是真正执行得顺序  如果变量被volatile修饰了 可以防止这种重排序  无论是单线程还是多线程 指令执行得顺序 就是我们编写得顺序

- volatile可以保证数据得可见性

  - 由于每个线程都有一个自己独立得工作内存,  同时还存在一个主内存 用于给多个线程共享变量得  每当有线程去修改共享变量 就会同步到主内存中   如果变量被volatile修饰之后 这样线程再去改变共享变量 会让其他线程得工作内存失效 强制刷新主内存得值    

    类似于static(是对象得共享)  volatile(保证线程中得共享)

| keepAliveTime   | 存活时间:  当线程池线程数超过了核心线程数时, 经过多长时间 回收这些多余的线程 |
| unit            | 时间单位: 比如:TimeUnit.SECONDS 表示秒                       |
| workQueue       | 工作队列:  也叫任务队列 用于存放 提交了 但是没有执行的任务 （执行一次会删除一个） |
| threadFactory   | 线程工厂: 用于帮你创建线程  相当于 帮我门new线程             |
| handler         | 拒绝策略:  当线程池、队列都满了   新任务会触发拒绝策略       |



### 线程池执行流程

- **判断核心线程池是否已满**

  - 若未满（当前线程数 < `corePoolSize`），则创建**核心线程**执行任务
  - 若已满，进入下一步

- **判断任务队列是否已满**

  - 若队列未满，将任务**加入队列**等待执行
  - 若队列已满，进入下一步

- **判断最大线程池是否已满**

  - 若当前线程数 < `maximumPoolSize`，创建**非核心线程**执行任务
  - 若已达`maximumPoolSize`，触发**拒绝策略**处理任务。

  



### 线程池核心线程数和最大线程数设置指标

程池的**核心线程数（corePoolSize）** 和**最大线程数（maximumPoolSize）** 是决定线程池并发能力的关键参数，其设置需结合**任务类型（CPU 密集 / IO 密集）**、**系统资源（CPU 核心数、内存）**、**任务特性（执行时间、并发量）** 等指标综合判断

- 核心线程数:
  - cpu比较密集的话（如数据计算、复杂逻辑处理）线程执行时 CPU 利用率高   一般设置为CPU核心数+1 左右把
  - IO比较密集的话  任务执行过程中大量时间用于等待 （如数据库查询、网络请求、文件 IO）CPU 利用率低    一般设置为CPU核心数*2
- 最大线程数:
  - cpu比较密集的话   一般设置为  与核心线程数**保持一致   ** 因为比较密集的话 无需创建额外临时线程
  - IO比较密集的话   一般设置 为CPU核心数*2  或者  *4   原因 在于应对突发任务峰值，当核心线程忙且队列满时，临时线程可快速处理积压任务



### 线程池的工作队列一般怎么选择

- ArrayBlockingQueue(数组队列):   一般情况下    任务数量可预估  或者 普通吞吐量(内存可控的情况下)   适用于 数组 队列 
- LinkedBlockingQueue(链表队列):  一般情况下  任务数量不可控  或者 高吞吐量   适用于 链表队列



### 线程池有哪些拒绝策略

- 默认策略  直接抛出异常  : `ThreadPoolExecutor.AbortPolicy()`
- 让调用者所在的线程来执行任务(谁提交的任务 谁负责执行): `ThreadPoolExecutor.CallerRunsPolicy()`
- 丢弃队列中最靠前的任务,并执行当前任务: `ThreadPoolExecutor.DiscardOldestPolicy()`
- 直接丢弃任务: `ThreadPoolExecutor.DiscardPolicy()`



### 什么是CAS

CAS是一种乐观锁机制，用于实现多线程环境下的无锁并发操作   主要的原理在于:**在更新一个值之前，先比较该值是否与预期值一致，若一致则更新为新值，否则不做操作并返回失败**



java中提供的 CAS 相关实现主要集中在`java.util.concurrent.atomic`包下

- 基本类型原子类
  - AtomicInteger : 对`int`类型进行原子操作（自增、自减、赋值等）
  - AtomicLong :  对`long`类型进行原子操作
  - AtomicBoolean  : 对`boolean`类型进行原子操作
- 引用类型原子类:
  - AtomicReference: 对任意类型的对象引用进行原子操作
  - AtomicStampedReference :  普通`AtomicReference`可能存在 ABA问题  (对象被修改为 B 后又改回 A，CAS 会误认为未修改 )    这种原子类解决  在对象引用之外，额外维护一个`int`类型的 “版本号”（stamp），更新时同时检查引用和版本号
- 数组类型原子类：
  - AtomicIntegerArray： 对`int`数组的元素进行原子操作





### IO流分类

- 根据流向: 
  - 输入流: 对文件的内容进行读取 
  - 输出流: 对文件的内容进行写入
- 根据单位:
  - 字节流: 每次读取或者写入一个字节的
  - 字符流: 每次读取或者写入一个字符的
- 根据功能:
  - 节点流: 可以从某个节点读取  向某个节点 写入数据  说白了就是普通输入和输出流
  - 处理流:  对已存在的流做二次封装 ，可以实现更加丰富流的处理方式 还可以提高读写效率   说白了就属于缓冲流



### 字节流和字符流区别

-	字节流: 可以对任何文件进行读写(图片 视频 音频 文件 doc),但是对于文本内容 读写可能出现中文乱码
-	字符流: 只能适合读写文本内容 可以防止出现中文乱码



### 缓冲流底层原理

可以提高输入输出流的读取和写入的效率,原理是每次读取完一定的字节或字符 不会立即写入 ,先放在缓冲(内存)中，等待缓冲区存储满了 一口气写入到本地磁盘,由于缓冲速度高于本次磁盘速度 所以提高了工作效率  同时减少了访问本地磁盘次数 提高了磁盘使用寿命



### 序列化和反序列化

###  阻塞IO和非阻塞IO的区别或BIO和NIO区别

- BIO(阻塞IO): 

  - 实现简单（无需轮询）；CPU 利用率高（阻塞时释放 CPU）
  - 并发低（一个连接占用一个线程，线程阻塞时闲置）；线程切换成本高

- NIO(非阻塞IO):

  - 不阻塞进程，可同时处理其他任务
  - 轮询消耗 CPU   实现复杂（需处理轮询逻辑)

### http协议和https协议区别

- 安全性不同: http不加密的数据,数据传输时时明文,https可以将数据进行加密传输   确保数据传输过程中是安全的   即使被拦截 也无法轻易解密
- 端口号:  http默认是80端口 ，https默认是443端口
- 性能不同:  http不需要加密 或接密 所以传输时速度时比较快, https要进行加密和接密  肯定会影响整个性能
- 适用场景不同:  http适合对安全要求不高的网站和信息 比如: 新闻, 分页页码数,搜索的内容,  https适合对用户隐私或者数据安全性很高的网站  比如:银行卡   支付密码  个人信息 身份证号



### TCP和UDP区别

- UDP协议是没有链接的,TCP是面向连接的
- UDP是不可靠的(只管发送 不管接受 类似于广播)
  TCP是可靠的(传输数据和接受数据不会丢失 都需要负责)
- UDP是面向报文传输(交给UDP无论是多长的报文,无需等待就可以数据包的形式进行发送)  
  TCP是面向字节流传输(才可以传输超文本内容,传输数据等待缓存区存储满了 一口气 以流的形式进行发送)



### TCP三次握手

- 第一次握手: 浏览器向服务器发送一个建立连接的请求(SYN请求连接数据包)
- 第二次握手: 服务器告诉浏览器 我同意你的连接请求 同时服务器也向浏览器建立连接的请求(ACK确认数据包+SYN请求连接数据包)
- 第三次握手: 浏览器也要告诉服务器 我同意你的连接(ACK确认数据包)



### TCP四次挥手

- 第一次挥手: 浏览器打算断开连接,向服务器发送一个FIN数据包
- 第二次挥手: 服务器接收到了对方发送的FIN数据包 就向浏览器发送ACK确认数据包 
- 第三次挥手: 服务器也打算断开连接, 向浏览器发送一个FIN数据包
- 第四次挥手: 浏览器接收了对方发送FIN数据包  向服务器发送ACK确认包



### 为什么握手需要三次 但是挥手需要四次

数据的发送和接收是独立进行的,服务器在收到客户端的 FIN 报文后，可能还有数据需要发送，因此先回复 ACK，等数据发送完毕后再发送 FIN 报文, 主要是为了确保数据完整传输



### OSI网络七层模型

- 物理层:  负责将数据转换为电信号、光信号或无线电信号(如网线、光纤、无线信道）传输原始比特流（0 和 1）
- 数据链路层: 将物理层的原始比特流封装成 “帧”   （MAC 地址   交换机（Switch）、网桥（Bridge））
- 网络层: 实现不同网络之间的数据包路由和转发 (如 IP 地址  )        路由器
- 传输层: 提供可靠的 “端到端” 数据传输服务  负责数据的分段、重组  （TCP和 UDP协议）
- 会话层: 建立、管理和终止通信双方的 “会话”
- 表示层: 处理数据的格式转换和表示   确保接收方能够理解发送方的数据  （ASCII  UTF-8）
- 应用层: 为用户应用程序提供网络服务  （http   https）

### SQL语句分类

- DDL: 数据定义语言,用于创建和删除数据库对象的命令(创建数据库  创建表  创建视图 创建索引 创建函数.... )  共同特性 `不能回滚事务 底层会自动提交事务`

  比如: create(创建)   drop(删除) alter(修改)  truncate(截断表(清空表数据))

- DML: 数据操作语言, 用于操作(增 删 改)数据库中表中数据的 

  比如:  insert(新增数据)  delete(删除数据)  update(修改数据)

- DQL: 数据查询语言, 这是最常用的命令(最难的) 用于查询数据库表中的数据

  比如:  select(查询数据)

- DCL: 数据控制语言，用于控制数据库对象的使用权限的 (控制具体哪个用户 可以操作哪个数据库 哪些表的权限)  --- Linux再讲

  比如:  grant(授权)    revoke(撤销权限)

- TCL: 事务控制语言, 用于对数据做提交事务(真正保存) 或者 回滚事务(撤销)的 主要是为了保证数据一致性....

  比如: commit(提交事务)   rollback(回滚事务)



### delete和truncate和drop区别

- delete: 可以用于删除表中数据,  而且可以添加条件来限定行数  是属于DML语言  可以进行事务回滚   不会释放空间（因为不是真删除而是打上删除的标记  等事务提交的时候 才是真正删除 才会释放空间） 不会删除表中定义(比如:表中定义的注释 约束  索引... 因为表还在 所以不会删除) 

  - 语法: `delete from 表 where 条件`

- truncate:  也叫截断表  用于清空表中数据   不能添加条件  是属于DDL语言   不可以进行事务的回滚   会释放空间   不会删除表中定义 

  - 语法:  `truncate table  表`

- drop: 可以用于删除表   不能添加条件   是属于DDL语言  不能进行事务的回滚  会释放空间   会删除表中的定义

  - 语法:  `drop  table 表`

  >drop(删除表)>truncate(先删除表再创建表) > delete(每个删除得数据都要删除标记 数据越多 效率越低)

### mysql约束

- 非空约束: not null   保证字段不能存储null
- 默认约束: default   保证字段一定有值  如果插入(包含null)了就失效了 如果没有插入 设置成默认值
- 唯一约束: unique  保证字段唯一性  但是数据可以存储多个null   并且唯一约束 会自动添加唯一索引 
- 检查性约束: check  用于给字段的数据添加一些 数据校验 (比如:控制性别 控制数据范围)  但是mysql 8之前 语法通过但是会失效 
- 主键约束: primary key(pk) 用于控制数据唯一性的方式   相比唯一约束  它主键约束会自带唯一和非空两种特性  并且每张表有且只有一个主键(联合主键也是一个主键)   它会自动添加主键索引
- 外键约束: foreign key(fk) 定义在具有父子关系的子表中，就是子表的外键 必须对应父表中的主键(或者唯一键)  目的是为了确保数据完整性的   并且外键字段 可以为null   还可以重复 



### mysql三范式

- 第一范式(1NF):  要求设计数据库表字段 具有原子性(不可分割)   就是不能将类似于 java中的对象 集合 数组 这类的数据当成字段

- 第二范式(2NF): 先满足1NF  又要求 其数据必须依赖于主键   说白了 就是让你每张表 添加主键   因为主键是唯一非空的 它可以防止数据冗余

- 第三范式(3NF): 先满足2NF 又要求 不能有其他表中的非主键列   就是表示不能有重复的列  这样就会出现很多数据冗余 说白了就是强调你 添加主外键关联

  >但是:第三范式在实际开发过程中是可以违反的,因为他是一种数据库优化的方案   因为以后有很多需求  可能会用到多张表的数据  如果使用频率特别高  每次查询都需要需要做多表关联 性能肯定会受到影响
  >
  >比如: 首页显示 员工姓名 工资 部门名称 会经常查询 经常用 每次查询肯定都需要查询员工表和部门表的两个数据  所以就违反了第三范式   可以给员工表添加一个冗余字段部门名称  这样虽然会浪费空间 多存了冗余数据  好处再于 每次查询 只需要查询一张表  以空间换时间提高了效率

### 内连接和左外连接区别

- 内连接: 只会把满足匹配条件的数据查询出来, 不匹配的数据无法查询出来  匹配条件也称之为关联条件
- 外连接: 
  - 左外连接: 把左边的表当成主表(无论是否匹配都会展示出来) 把右边的表当成复表(只显示满足的数据)  不匹配的数据会显示null
  - 右外连接: 把右边的表当成主表(无论是否匹配都会展示出来) 把左边的表当成复表(只显示满足的数据)  不匹配的数据会显示null
  - 全外连接: mysql不支持   表示左右两边的表都是主表  两边的表都会全部展示

### 什么是索引

索引 是用来加快查询效率的一种方式，可以不断缩小 查询数据的范围   从而节省了 查询的时间   类似于字典  只需要 找到  哪个范围的数据   而不是像原来一样 逐一的查询   底层实现 是通过B+树 实现的 



### 索引分类

- 普通索引: index
- 唯一索引: 

  - 主键索引: primary key 添加主键约束 自动添加
  - 唯一索引: unique  添加唯一约束 自动添加 

- 全文索引: fulltext 用于搜索长篇文章 比较合适

- 联合索引:

  - 联合普通索引: index(name,age)
  - 联合主键索引: primary key(sid,cid)
  - 联合唯一索引: unique(name,phone)



###  如何查看是否走了索引

通过explain+查询语句  可以查看一个查询语句的执行计划   就可以看到整个sql语句整体性能和是否走了索引

比如: explain select * from 表;

```sql
select_type: 表示是否有多表关联或者子查询 或者union
table: 这条查询语句 使用表有哪些(表名,别名,derived(虚拟表))
type: 可以用于查看sql语句整体性能  也可以查看是否走了索引, sql语句为了表示性能 主要分为不同的级别
--  system>const>eq_ref>ref>
--  fulltext>range>index>all
--  system: 表示只有一行数据 或者空表  但是InnoDB引擎 都是all
--  const: 表示使用主键引用或者唯一索引
--  eq_ref: 用于连接查询 一个表只返回一条记录 并且这行记录 正好式另一个表的主键或唯一键 但是不能为null
--  ref: 相比上面 没有这么多要求 返回的是不唯一的 等值查询 (普通索引)
--  fulltext: 全文索引, 如果全文 和普通索引 都有 优先使用全文
-- range: 索引扫描范围  常用> <  between and  like
-- index: 索引全表扫描  把索引从头扫描一遍
-- all: 全表扫描
possible_keys和key: 两个key都是类似于的  一般来说如果没有走索引 一般是null  如果走了索引 通常都会有值表示索引名称  而且两者相同
rows: 执行查询语句的时候  估算的行数  可能不是精确的值
```





### 索引是否越多越好

不是,  原因是如果给每个字段都添加索引,确实可以提高查询效率,但是会降低增 删 改的效率, 因为每次修改 都需要更新索引值 



### 什么字段适合添加索引

- 经常作为查询条件的字段 
- 很少修改(增删改)的字段 需要维护索引
- 经常作为主外键关联的字段
- 值不重复 并且不为空的字段  因为空会造成索引失效
- 经常排序  分组的字段  



### 最左匹配原则

最左匹配原则主要表示使用了联合索引,如果查询时 条件涉及到了索引的多个字段  只有联合了索引最左边的字段  它匹配 才能重复利用索引

>面试题2: 有一个表 创建了联合索引(A,B,C)   如果 查询时 针对于B 和 C作为条件  是否会走索引  
>
>答案: 不会  先匹配A  才能匹配B 
>
>面试题3: 有一个表 创建了联合索引(A,B,C)   如果 查询时 针对于A 和 B作为条件   是否会走索引 
>
>答案:  会  匹配A的数据 
>
>--- 联合索引(A,B,C)
>
>--- 匹配(A)  匹配(A,B)   匹配(A,B,C)   都会走索引





### 索引失效

- 使用or: 它是或者  不一定所有的条件 都添加了索引
- 使用like做模糊查询   使用前% 不走索引  后%走索引  原因在于前% 包含了0-n个字符 不知道 从谁开始匹配
- 使用 is null 或者 is  not  null  不走索引
- 使用 不等于 != 或者 <> 不走索引 
- 如果条件字符串内容,没有添加引号  mysql会自动进行类型   类型都变了  也不会走索引
- 索引的字段  参与了运算  不走索引
- 索引的字段 使用了函数 也不会索引



### 为什么mysql采用B+树作为索引 而不是 使用 数组  链表  哈希表 二叉树  平衡二叉树 红黑树

- 数组: 数据库无论是查询要考虑性能  增删也要考虑性能 如果采用数组  因为它增删效率很慢 所以不适合

- 链表: 增删比较快 查询比较慢 更不适合索引

- 哈希表:  底层虽然兼容 数组和链表的特点   优点查询和增删 再不考虑哈希冲突下 时间复杂度都是O(1)  但是缺点  它是没有顺序的   而数据库经常做范围查询 比如:>  < between   无法实现正常的范围查询 

- 二叉树: ....

  - 根节点 : 属于树的最上层的节点   只有一个

  - 子节点:  除了根节点 它的下面还连接其他的节点 也会分左右(左子树  右子树)

  - 叶子节点:  下面没有连接其他节点  属于树的最下层

    >- 二叉树 查找元素的规则:  首先先和根节点比较 ,如果不相等 , 查找的元素 大于根节点    继续右子树去比较  反之 继续和左子树比较  ...
    >- 二叉树 插入元素的规则:  首先判断根节点 是否为null  如果是null插入根节点   如果不是null 去比较大小  如果插入的元素>根节点   放入右子树(如果它也不是null   继续去比较大小  )   如果插入的元素<根节点  放入左子树

- 平衡二叉树:  任何节点 的两个子树  高度差 不超过1 就是平衡的   而且两个左右子树 也是独立平衡二叉树

  >二叉树 还是平衡二叉树 虽然查询效率比较高 而且还可以做范围查询  因为它有顺序   但是如果出现一些极端情况 插入的数据 都是递增的数据   由于树的特点只会插入到右子树   这样就会树的高度越来越高  类似于链表了  会影响查询性能   它不适合做索引



### B-树和B+树有什么区别

- B-树: 底层就是一个平衡二叉树   区别在于每个节点 存储的很多内容(主要包含: 主键 , 指向下个节点的指针, 除了主键的其他数据)  优点在于 可以降低树的高度  缺点在于  每个节点都存储了数据(data)  无法减少IO读取磁盘的次数  所以也不适合做索引
- B+树:  它其实是B-树的一个升级版  同样降低树的高度    区别在于 每个节点(只存储 主键 和 指向下一个节点的指针) 而所有数据 都存储在叶子节点上   这样就可以减少IO 读取磁盘的次数   并且每个叶子节点  是可以连接在一起的   这样就会形成一个有序的链表   这样就更加适合做范围查询  所以 mysql采用B+树做索引



### 什么是视图

视图(view) 是一种虚拟表(本身不存在的) 只会存在一条查询语句   每次执行视图时候  相当于把你们保存的查询语句重新执行一遍



### 视图优缺点

- 优点: 
  - 复用 : 视图创建好了 可以重复使用
  - 安全:  使用视图的用户  只能访问 被允许的查询字段  具体的表中逻辑无法查看
  - 简单:  对于使用视图   无需考虑 里面封装了什么复杂的sql语句
- 缺点: 视图不能提高查询效率   因为使用视图  都会重新执行里面的SQL语句   而且表中的数据 还是存储在原来的位置   最多提高了 使用的效率



### 什么是事务

事务就是绑定在一起的逻辑工作单元(功能) 这个工作单元 可能包含多次的sql语句的操作 是为了保证这么多sql语句要么都成功 要么都失败   一般用于处理 非常复杂 的业务逻辑



### 事务四大特性ACID

- 原子性：绑定在一起的这些sql语句 不能分割  要么都成功 要么都失败
- 一致性:  数据在运行的前后 结果是保持一致的
- 隔离性: 一个事务和其他事务之间  是不能相互干扰的
- 持久性: 事务一旦提交了  对数据修改永久的 



底层实现原理

- **原子性**：Undo Log 负责回滚，确保操作要么全成，要么全退。
- **一致性**：由原子性、隔离性、持久性及数据库约束共同保障，是最终目标。
- **隔离性**：通过锁机制控制并发修改冲突，通过 MVCC 实现高效的非阻塞读，不同隔离级别对应不同的锁和 MVCC 策略。
- **持久性**：Redo Log 确保已提交的修改在崩溃后可恢复，是数据不丢失的核心保障。





### 事务并发问题

- 脏读: 事务A读取了事务B 更新了 但是没有提交的数据， 突然事务B 进行了回滚事务  这样事务A读取的数据就是脏数据
- 不可重复读:  事务A多次读取同一个数据 , 事务B在 事务A读取过程中  对数据做了更新并且提交事务  导致事务A多次读取同一个数据不一致
- 幻读: 事务A多次读取同一个数据 , 事务B在 事务A读取过程中  对数据做了新增或删除并且提交事务  导致事务A多次读取数据行数不一致



### 事务隔离级别

| 隔离级别 |   脏读   | 不可重复读 |   幻读   |
| :------: | :------: | :--------: | :------: |
| 读未提交 | 无法解决 |  无法解决  | 无法解决 |
| 读已提交 |   解决   |  无法解决  | 无法解决 |
| 可重复读 |   解决   |    解决    | 无法解决 |
|  串行化  |   解决   |    解决    |   解决   |

==注:mysql默认隔离级别:可重复读,oracle默认:读已提交   因为数据库隔离级别越高 数据安全性越好 但是性能就会越差, 而数据库 不仅仅考虑安全性  它的性能也需要考虑    所以数据库会采用这中方案(不是最高的 也不是最低的)==

### 聚簇索引和非聚簇索引区别

- 聚簇索引
  - 创建聚簇索引后，数据会按照索引键的顺序重新排列
  - 通常默认基于主键创建。如果表没有主键，数据库会自动生成一个隐藏的聚簇索引 比如row_id
  - 范围查询效率极高。比如 “查询 id 在 100-200 之间的用户”，因数据物理连续，只需定位到起始位置后顺序读取即可
  - 适合场景: 优先用于 “主键” 或 “高频范围查询” 的字段，如用户表的`id`、订单表的`create_time`

- 非聚簇索引

  - 索引与数据分离。索引是单独的 B + 树结构，叶子节点只存 “指向数据的指针”
  - 查询需 “回表”。通过非聚簇索引找到聚簇键后，需再通过聚簇索引查找实际数据，多一次 IO 操作
  - 适合高频筛选字段。比如对 “用户性别”“订单状态” 等高频查询字段建非聚簇索引，能快速缩小查询范围
  - 适合场景: 用于 “高频过滤但非范围查询” 的字段，如用户表的`phone`（查指定手机号用户）、商品表的`category_id`（查某分类商品）。

  


### mysql8的窗口函数

### mysql中MVCC底层原理

MySQL 中的 MVCC（多版本并发控制）是 InnoDB 存储引擎实现隔离级别的核心机制，其核心思想是通过保存数据的多个版本，实现读写不冲突，从而提高并发性能

- MVCC 解决了读写冲突问题:
  - 读操作（SELECT）不需要加锁，直接读取历史版本数据。
  - 写操作（INSERT/UPDATE/DELETE）只锁定当前版本数据，不阻塞读操作
  - 不同事务可同时读写同一行数据，通过版本控制实现隔离
- 底层结构实现
  - 隐藏列
    - **DB_TRX_ID**：记录最后一次修改该行的事务 ID（6 字节）
    - **DB_ROLL_PTR**：回滚指针（7 字节），指向该行的历史版本（存储在 undo log 中）
    - **B_ROW_ID**：若表无主键，InnoDB 自动生成的行唯一标识（6 字节）
  - undo log(回滚日志)
    - 用于保存数据的历史版本，是 MVCC 实现多版本的基础
    - 当事务修改数据时，旧版本数据会被写入 undo log，通过 **DB_ROLL_PTR** 串联成版本链
  - Read View(读视图)
    - 事务执行查询时生成的 “快照”，用于判断哪些版本的数据对当前事务可见
    - 包含 4 个核心字段
      - `m_ids`：当前活跃（未提交）的事务 ID 集合
      - `min_trx_id`：`m_ids` 中最小的事务 ID
      - `max_trx_id`：当前系统尚未分配的下一个事务 ID（即最大可能的事务 ID+1）
      - `creator_trx_id`：当前事务自己的 ID
- 版本可见性判断规则 : 事务通过 Read View 检查数据版本的可见性
  - 若数据版本的 `DB_TRX_ID` = 当前事务 ID（`creator_trx_id`）：可见（自己修改的）
  - 若 `DB_TRX_ID` < `min_trx_id`：可见（修改该版本的事务已提交）
  - 若 `DB_TRX_ID` > `max_trx_id`：不可见（修改该版本的事务在当前事务之后启动）
  - 若 `min_trx_id` ≤ `DB_TRX_ID` ≤ `max_trx_id`：
    - 若 `DB_TRX_ID` 在 `m_ids` 中：不可见（事务未提交）。
    - 若不在 `m_ids` 中：可见（事务已提交）
- 针对于不同的隔离级别MVCC行为 : MVCC 在不同隔离级别中通过 Read View 的生成时机区分
  - **读未提交（Read Uncommitted）**：不生成 Read View，直接读取最新版本（可能看到未提交数据）。
  - **读已提交（Read Committed, RC）**：每次执行 SELECT 时生成新的 Read View（只能看到已提交的数据）。
  - **可重复读（Repeatable Read, RR）**：仅在事务第一次 SELECT 时生成 Read View，后续查询复用（保证事务内多次读取结果一致）。
  - **串行化（Serializable）**：不依赖 MVCC，通过加锁实现完全隔离。
- MVCC与锁的关系: 
  - **读操作**：通过 MVCC 读取历史版本，无需加锁（快照读）
  - **写操作**：需加行锁（如排他锁），防止并发修改，同时生成新版本并写入 undo log
  - 特殊读操作（如 `SELECT ... FOR UPDATE`）会加锁（当前读），读取最新版本







### mysql中行锁和表锁区别

| 维度     | 行锁（InnoDB）             | 表锁（如 MyISAM、InnoDB）   |
| -------- | -------------------------- | --------------------------- |
| 锁定范围 | 单行（或符合条件的多行）   | 整个表                      |
| 并发性能 | 高（冲突少）               | 低（冲突多）                |
| 适用场景 | 高并发写操作               | 读多写少、全表操作          |
| 死锁风险 | 可能发生                   | 几乎不发生                  |
| 依赖索引 | 是（无索引退化为表锁）     | 否                          |
| 事务支持 | 支持（与事务生命周期绑定） | 不依赖事务（MyISAM 无事务） |



### mysql中回表查询是什么

在 MySQL 中，**回表查询**是 InnoDB 存储引擎在使用二级索引（非主键索引）查询数据时，由于二级索引无法直接提供查询所需的全部字段，需要通过二级索引记录中的主键值再次访问主键索引（聚簇索引）用于获取完整行数据

- **聚簇索引（主键索引）**
  - 叶子节点直接存储**完整的行数据**，整个表的数据按照主键顺序组织（一张表只有一个聚簇索引）
- **二级索引（非主键索引，如普通索引、联合索引）**
  - 叶子节点仅存储**索引字段值 + 主键值**，不包含完整行数据。二级索引的作用是快速定位到主键，再通过主键查询完整数据
- **回表查询触发流程**: 当查询使用二级索引，且需要的字段**超出二级索引包含的信息**（即二级索引未覆盖所有查询字段）时，会触发回表
  - 先通过二级索引查询，找到匹配记录的**主键值**
  - 再通过主键值访问**聚簇索引**，获取完整的行数据



### sql注入

sql注入:一般是前端可以传递一些`非法参数`  最后通过`字符串拼接`的方式 来处理sql语句  最终执行sql语句 可能执行效果 不是按照程序员预期执行  达到了服务器的目的  比如: delete from 表 where name='随便写'  or 1=1  这里or 1=1 就是非法参数  就会造成明明只是删除一条  最后删除全部数据

- 解决方案(预编译): 使用预编译对象PreparedStatement来解决 ,它会先编译sql语句  目的是让sql语句结构 是固定的, 并且sql语句的参数 是通过? 做占位符   同时还可以实现一次编译 多次运行  执行效率也会高一些 
- 解决方案(前端): 对于所有用户输入数据 进行验证  比如做js验证 



### PreparedStatement和Statement区别或者mybatis中#{}和${}区别

### 什么是ThreadLocal

ThreadLocal是java中的一个特殊的类,用于再多线程的环境下 维护线程的局部变量 ,一般情况下 如果多个线程共享同一个变量 可能引发线程安全的问题,而ThreadLocal可以为每个线程提供一个独立的变量副本,每个线程都可以独立修改这个副本 不会影响到其他线程对象的副本   适合再多线程下共享资源时使用： 比如: 存储数据库连接，Session会话管理....    常用方法如下:

- set(): 设置当前线程的变量副本
- get():  获取当前线程的变量副本
- remove(): 清除当前线程的变量副本



### http请求方式

- post请求: 用于向服务器提交数据(新增), 通常是通过form表单设置请求方式为post  和 一些测试软件(postman  jmeter)
- get请求: 用于获取服务器数据(查询)   应用最广泛的请求方式   比如:  地址栏输入网址 ,超链接 , css ,js , 图片 ...
- put请求: 用于修改服务器数据

### get请求和post请求区别

- 应用场景: get请求一般用于获取服务器数据, post请求一般用于向服务器提交数据
- 数据传输方式: get请求传递的数据会在地址栏显示， post请求属于隐式提交  可以查看浏览器控制台 
- 传输长度:  get请求由于在地址栏显示 不能无限拼接  根据浏览器的限制决定 不同的浏览器长度限制不同  post请求 理论上没有长度限制
- 安全性：post方式相比get方式更安全
- 如果上传文件 只能使用post方式



### http请求状态码

- 200: 请求成功
- 301: 永久重定向   永久移动，请求的资源已被永久移动到新的位置，应使用浏览器缓存的HTTP_REFERER字段或更明确的条件将用户重新导向新的URL。
- 302:  临时重定向   临时重定向得意思，请求的资源临时移动到了新的位置，浏览器应尝试使用新的URL获取资源
- 304:  表示客户端发送的请求资源未被修改,服务器端无需返回资源内容
- 400: 客户端参数数据接收异常
- 403: 请求被拒绝(没有权限)
- 404: 一般是访问的资源找不到  或者访问了web-inf资源
- 500: 服务器出现异常
- 501: 这个状态码表示服务器无法识别这个请求方法或者请求方法没有被实现
- 502: 错误网关 — 服务器接收到来自上游服务器的无效响应
- 503: 无法获得服务



### servlet生命周期

- 初始化:  servlet`默认第一次访问`先实例化(Servlet对象)  然后调用`init() 做初始化` 而且只执行`一次 ` ,但是会出现首次延迟的问题 所以可以修改配置文件添加`load-on-startup`标签让其再服务器启动执行
- 请求处理: 每次发送请求都会进入`service()` 来进行处理 `执行多次  `
  - 如果HttpServlet实现的Servlet请求入口也是service(),底层会根据请求方式去判断 然后调用doGet()  doPost()...
- 销毁: 服务器关闭了 执行`destroy() `进行销毁 `执行一次`



###  转发和重定向区别

- 请求次数不同: 转发只会发送一次请求 , 重定向会发送两次请求(第一次是302  第二次浏览器重新发送一个get请求 表示访问的最终地址)
- 地址栏是否会发送改变: 转发地址栏不会发生改变 ,重定向地址栏会发生改变 
- 是否共享request数据: 转发由于只有一次请求 可以共享request数据  但是重定向是两次请求 不能共享request数据
- 应用场景不同: 转发是服务器内部跳转 只能访问服务器内部资源 同时还可以访问WEB-INF资源,  重定向 可以访问外部资源 也可以访问内部资源 但是不能访问WEB-INF资源

### springmvc工作流程

- 前后端不分离

  >1.请求先到核心控制器(DispatherServlet)
  >
  >2.核心控制层查询请求映射器(HandlerMapping) 来看是否存在  如果不存在 返回404  如果存在返回对应地址得Controller
  >
  >3.核心控制器再通过代理对象(HandlerAdapter)动态调用Controller中得对应得Method方法
  >
  >4.控制层(Controller)开始执行方法  处理请求   返回是ModelAndView对象给核心控制器
  >
  >5.核心控制器再调用视图解析器(ViewResolver)对其进行解析   解析成Model对象和View对象  返回给核心控制器
  >
  >6.View负责将Model实现给对应得客户端

- 前后端分离

  >1.请求先到核心控制器(DispatherServlet)
  >
  >2.核心控制层查询请求映射器(HandlerMapping) 来看是否存在  如果不存在 返回404  如果存在返回对应地址得Controller
  >
  >3.核心控制器再通过代理对象(HandlerAdapter)动态调用Controller中得对应得Method方法
  >
  >4.控制层(Controller)开始执行方法  处理请求   Json对象给核心控制器
  >
  >5.核心控制器直接将Json数据返回前端

### 同步请求和异步请求区别

- 同步请求: 发送请求后需要等待服务器响应，这个期间会被阻塞,只有等待服务器响应后 上面的代码全部执行完了 才可以执行后续的内容 (类似于单线程执行)
  - 应用场景: 适用于需要立即响应结果的场景，并且 后续的操作 需要依赖于这个结果  比如: 登录后才知道访问哪个页面,  前后不分离 存储作用域  如果不存储完 不可以展示数据 
- 异步请求: 发送请求后 不需要等待服务器响应  就可以执行后续代码  最后什么时候响应 再通过回调函数 去更新对应的数据（类似于多线程）
  - 应用场景: 适用于不需要立即响应的场景   比如:订单支付 (不支付 也不影响去浏览其他商品  支付成功了才会发货)   视频播放 点赞 评论    前后端分离项目全是异步请求   RabbitMQ异步请求

### 什么是ORM

ORM: Object  Relation Mapping 对象关系映射，对象就是java中的对象泛指的是实体类,关系 就是表示关系型数据库，映射就是表示一一对应 , ORM其实就是java中的对象 和 关系型数据库的一条数据 做了一一对应  目的:将数据库对于数据的操作转换成java对于对象的操作  比如: 新增一条数据  就可以表示为新增一个对象  修改一条数据 表示修改一个对象

- 实现方式:  实体类 和 表  需要一个第三方 设置了哪个类对应了哪张表   哪个的字段对应了类中的哪个属性 而这个第三方文件 称之为映射文件

### mybatis一级缓存和二级缓存

- 一级缓存: 是sqlSession级别的缓存,会再同一个sqlSession中有效的， 默认开启的   无需多余配置    对数据做增删改 会自动清除缓存 防止脏读
- 二级缓存: 是Mapper级别的缓存, 可以再不同的sqlSession中有效的  默认关闭的  需要手动配置  对数据做增删改 会自动清除缓存 防止脏读    只有一级缓存失效了 才会使用二级缓存



### Mapper接口的工作原理

Mapper接口就是原来的dao接口,都属于数据访问层,是根据映射文件namespace属性 对应哪一个映射文件， Mapper接口的方法 的名称 对应映射文件中的标签id  这样实现数据库操作只需要再标签中写一条sql语句即可.

- 工作原理:  先 根据Mapper接口的全类名 + 方法名 作为一个key值  就可以快速定位到哪个映射文件中的哪个标签，底层实现是根据jdk动态代理方式  运行时会为每个mapper接口创建一个对象Proxy(代理对象),会拦截Mapper接口的方法 ,拦截之后 帮你动态调用哪个映射文件的标签   进而就可以执行里面的sql语句  并且运行的结果 也会拦截   然后根据方法的返回值 进行自动封装返回 

- mybatis主要提供了几种拦截器

  - (3)执行器拦截器:  拦截sql语句 用于执行sql语句的  可以实现事务的   还可以用来实现缓存....
  - (1)参数拦截器:  用来拦截传递的参数   或者 用来做 转换使用
  - (4)结果集拦截器:  用于拦截查询的结果   处理返回成方法对应的结果    
  - (2)sql语句构建拦截器: 用于修改sql语句的  添加#{}参数 替换成占位符   或者添加${}参数 换成 拼接字符串 

- Mapper接口方法可以重载吗?

  不能,  因为mapper接口是根据全类名+方法名 用于匹配哪个映射文件的哪个标签  而重载 它是方法名相同 参数不同   所以如果重载了 就会出现重名的方法   就会出现很多相同的key  就无法对应到哪个标签了



### 乐观锁和悲观锁

再项目中经常会出现数据并发的问题，多个用户同时访问同一个数据 并且对其进行修改 就可能出现数据不一致的问题 这样mybatis就可以通过`乐观锁`和悲观锁来解决这些并发问题

- 乐观锁: 给表添加一个字段 版本号(int) 或者 时间戳(timestamp ) 两种字段方式都一样  只是类型不相同 , 每次更新 先查询一下版本号 或者时间戳  等到用户修改完毕后 提交事务之前 再来匹配版本号   如果两次结果是一致的  说明没有被别人修改过  就可以提交事务  同时还需要更新版本号(+1)或者时间戳(最新时间)  如果两次结果不一致  说明被其他人修改过  则不能提交事务 就回滚事务
- 悲观锁: 是数据库控制的  我们可以再查询时 添加for update就会锁住查询的数据   这样其他用户无法操作这些加了锁的数据  只有事务提交了  才会释放锁  这样其他用户才可以操作  类似于Synchronized 



### 什么是spring

spring是一个轻量级的`控制反转(IOC)`和`面向切面编程(AOP)`的容器框架, 主要用来管理对象的生命周期的，就可以更加方便 控制很多对象的资源消耗



### 什么是IOC

IOC就是控制反转,就是将创建对象控制权  交给spring容器管理，这样spring就可以集中管理 哪些对象只需要创建一个(单例),哪些对象需要创建多个   这样可以节省资源的消耗   那么这种控制方式 对象控制权原来是自己 后面交给spring容器  所以对象的控制权由主动 变成了被动 就是控制反转   底层是通过DI依赖注入实现的



### 什么是DI依赖注入

DI就是依赖注入,类和类依赖关系  不用自己赋值 ,交给spring它会帮你自动赋值(注入)，  注入时推荐使用接口类型来做依赖属性, 因为接口可以提供很多实现类, 换任意一个实现类 都是可以同一个接口类型  spring就可以很方便的切换不同的实现类   这样就可以降低各层的依赖关系  实现了解耦合 



### 依赖注入方式

- set注入: 在类中给依赖属性 添加set方法 ,并且spring配置文件的Bean添加Property 用来给依赖属性赋值  底层调用set()来完成赋值    是在对象创建之后 再通过set方法进行注入   并且注入 顺序和 个数 是可选的   所以set注入更加适合复杂依赖关系的建立  更加直观 
- 构造方法注入:   在类中添加一个有参构造方法，并且再spring配置文件 Bean添加constrcutor-arg标签  来配置有参构造的参数  底层走的有参构造实现的  是再对象创建时 完成属性的注入   而且注入属性一般是固定的(类型 和个数 )  所以就不太适合复杂依赖关系的建立
- 注解注入:  借助于IOC扫描注解 和 DI依赖注入注解 来完成
  - IOC注解:  @Controller  @Service  @Repository  @Component
  - DI注解: @Autowired    @Resource





### spring管理的bean生命周期

spring默认管理的bean都是单例模式,无论使用多少次都是一个对象，除了默认的模式 还提供很多种不同的生命周期

- 单例:  默认值 ,   读取spring配置文件(服务器启动)  就会加载所有需要的bean  创建出来      服务器关闭时 就可以销毁...

- 多例:  一个类spring使用一次 创建一个新的对象  类似于new对象 ， 所以每次使用Bean  spring就会开始创建  使用完后 自动回收(GC)

- request请求: 一个类spring每次接收到请求会创建一个对象

  请求处理完了  就可以销毁对象了

- session会话: 一个类spring每次会话才创建一个对象     会话超时了 或者强制销毁了  对象才会销毁 

- 全局session: application  属于应用级别 .... 





### 循环依赖

循环依赖就是两个类比如:A类 依赖于B 类    B类也依赖于A类, 这样如果采用构造方法注入  会再对象实例化完成对象依赖关系的建立  但是A需要B先创建      B需要需要A先创建    就造成了循环依赖的问题

- 解决方案: 
  - 可以重新设计   不设计成相互依赖的情况
  - 如果不能重新设计  可以采用set注入或者注解注入方式   因为他们都不会再bean创建时 构建依赖关系
  - 也可以添加@Lazy延迟加载注解  这样依赖属性不会立马注入 只会注入一个代理对象  只有它首次使用时 才会完成对象实例化过程 进行注入..





### spring常用注解

- IOC扫描注解:  一般使用再类上面的 ，用于标注不同的层的代码, spring配置文件 必须提供扫描包， spring只要扫描到类上添加了这类的注解  它就自动创建该类的对象 (底层等价于帮你自动编写了bean标签) 下面这个四个注解底层实现都是一样的  只是标注身份不同   默认bean的id: 默认类名首字母小写,如果想自定义id   @IOC注解("自定义的id")

  - @Controller: 用于标注控制层的注解，springmvc只有控制层允许接收请求
  - @Service: 用于标注业务层的注解,只有业务层可以正常完成事务功能
  - @Repository: 用于标注dao层的注解, 后期可以不写  因为是Mybatis使用的mapper接口 需要通过spring动态创建mapper接口实现类  不是靠扫描创建的 
  - @Component:   用于标注其他层的注解  比如:过滤器 监听器 拦截器  配置类....
- DI依赖注入注解:  一般写再类中成员变量上,前提是spring容器必须存在对应类型的bean对象
  - @Autowired:  是spring提供的注解, 自动根据spring容器中的bean类型去匹配   如果匹配上了 自动赋值  如果匹配不上  再根据依赖属性名和bean的id进行匹配   如果匹配上 也会自动赋值     如果根据类型匹配 出现多个相同的类型  它就不清楚 该注入哪个对象 出现异常
    - 解决方案: 
      - 可以修改关联属性名 让其和bean的id相同即可
      - 也可以添加@Qualifier("bean的id")
  - @Resource: 是java自带的注解, 也可以先根据类型匹配对象   如果发现多个相同的类型   也会出现异常    也可以修改关联属性名 和bean 的id去匹配   如果一致都可以自动注入    如果关联属性名 和bean的id不同 还有很多相同的类型  就可以修改@Resource(name="bean的id")
- AOP注解:
  - @Aspect : 配置切面
  - @PointCut: 配置切入点
  - @Before:  前置通知
  - @AfterReturning: 后置通知
  - @AfterThrowing: 异常通知
  - @After: 最后通知
  - @Around: 环绕通知
- MVC注解: 泛指springmvc注解

  - @RequestMapping  @GetMapping  @PostMapping...
  - @PutMapping  @DeleteMapping
  - @RequestBody: 前端提交json数据 后端转换成java对象
  - @ResponseBody:  方法的返回结果 不是地址 是响应结果 并且把响应结果转换成json
  - @DateTimeFormat: 前端提交字符串数据时  可以实现转换指定日期格式
  - @JsonFormat:  通过json返回给前端数据 包含了日期格式  默认返回的毫秒数   它可以修改转换格式

- 其他注解:

  - @Scope: 给bean对象设置作用范围  比如:@Scope("prototype")设置多例模式 
  - ...



### @Autowired和@Resource 两种注解的区别?

- @Autowired:  是spring提供的注解, 自动根据spring容器中的bean类型去匹配   如果匹配上了 自动赋值  如果匹配不上  再根据依赖属性名和bean的id进行匹配   如果匹配上 也会自动赋值     如果根据类型匹配 出现多个相同的类型  它就不清楚 该注入哪个对象 出现异常
  - 解决方案: 
    - 可以修改关联属性名 让其和bean的id相同即可
    - 也可以添加@Qualifier("bean的id")
- @Resource: 是java自带的注解,先根据名称匹配   再根据类型匹配对象   如果发现多个相同的类型   也会出现异常    也可以修改关联属性名 和bean 的id去匹配   如果一致都可以自动注入    如果关联属性名 和bean的id不同 还有很多相同的类型  就可以修改@Resource(name="bean的id")



### 什么是AOP

aop就是面向切面编程,主要用于将项目的通用功能(事务 日志 资源回收  异常处理  权限... ) 和 主要的业务功能 进行分离 ， spring就可以将这些通用功能统一处理 形成一个切面   而主要的业务功能只要经过这个切面  就会自动完成这些通用功能    对于开发者  只需要关注 业务逻辑 即可     底层实现jdk动态代理.   可以再不改变源程序的基础上  做增强处理





### AOP通知类型

- 前置通知: 是再目标方法执行之前调用 (aop:before标签 @Before注解 )

- 后置通知: 是再目标方法执行之后并且正常返回时执行(aop:after-returning标签  @AfterReturning注解)

- 最后通知: 是再目标执行之后调用 无论是否有执行总是执行的类似于 finally的位置 (aop:after标签   @After注解)

- 异常通知: 是再目标方法 执行时 出现异常 才会调用(aop:after-throwing标签    @AfterThrowing注解)

- 环绕通知: 是再目标方法执行 前后 都会执行   相当于前面4种通知的集合 (aop:around标签   @Around注解)



### spring如何做事务

- 声明式事务: 通过配置文件 编写spring事务管理策略  可以控制哪些方法 可以通过不同的事务的传播特性来做事务 
- 注解式事务: 再类上添加(表示类下面所有的方法都做事务) 或者 再方法上添加(表示当前方法需要做事务) @Transactional注解



### spring事务的传播特性

spring一共定义了七种传播特性，通过它可以 哪些方法 支持 事务 哪些不支持事务  哪个支持嵌套事务

- required: 默认值 必须要有一个事务   如果不存在事务 就会开启一个新事务
- requires_new:  新的事务  必须运行再自己创建事务中
- supports: 支持事务   不要求有事务  但是有也支持
- not_supported: 不支持事务   有事务也不会运行
- never: 永不支持事务    有事务抛出异常
- mandatory: 必须要有事务   没有事务抛出异常
- nested: 嵌套事务   可以支持多个事务之间 嵌套在一起  里面的事务 不会影响外层的事务



### spring事务注解@Transactional什么时候会导致事务失效

- 业务层对象可能没有被spring管理  比如:忘了写@Service
- 方法通过final  或者 static修饰   无法被重写 或者继承  会导致动态代理无法正常生成代理类   导致事务失效
- 比如mysql如果设置了MyISAM引擎 由于它不支持事务 也会导致事务失效
- spring aop只能对public的方法生成代理  所有如果使用了其他修饰符  事务也不会生效
- 事务传播特性设置不合理  比如:设置not_supported 或 never 都不支持事务  也会导致事务失效

###  前端和后端如何实现数据传递

- 首先前端访问页面  输入需要发送的数据   通过axios发送异步请求    将传递的数据 通过json格式传递给后端

  - @RestController 相当于具有@Controller功能  同时每个方法都会默认添加 @ResponseBody

### springboot全局异常映射

全局异常处理可以通过 `@ControllerAdvice`注解配合其他异常处理注解来实现，能够统一捕获和处理应用中抛出的异常    主要 设计就是2-3个注解

- **`@ControllerAdvice`**用于定义全局异常处理类，可捕获所有 `@Controller` 层抛出的异常
- **`@RestControllerAdvice`**：是 `@ControllerAdvice` + `@ResponseBody` 的组合，适合 REST API 项目，返回 JSON 格式的异常信息
- **`@ExceptionHandler`**：指定当前方法处理哪种类型的异常



### 什么是跨域

跨域简称cors(Cross-Orign-Resource-Sharing) 叫跨域资源共享, 再项目开发过程中 会发生很多网络请求  如果请求 地址中(协议、域名(ip地址)、端口  ) 只要有一个不同  就会产生跨域问题 ,这样项目中就 无法正常进行交互   特别是前后端分离的项目  一定会出现跨域问题   

- 后端解决方案 ：
  - 可以使用跨域注解@CrossOrigin 使用不是很灵活  只能控制所有的请求 或者某些单一的请求 进行跨域
  - 可以通过springboot实现跨域过滤器  是目前比较主流的方式  因为更加更加灵活来控制不同的 不同项目来源  不同的方式... 进行跨域

  - 可以通过springboot实现`配置类` 来进行跨域   目前也是比较主流的方案
- 前端解决方案 :
  - 通过vue  编写proxy 写一个代理组件  让其转发给后端

### springboot启动流程

springboot启动流程主要分为初始化、配置加载、容器启动、自动装配、组件初始化和启动这么几个阶段

- 初始化:  首先通过启动类注解@SpringbootApplication 下面的main方法 作为 启动起点, 通过调用SpringApplication.run()来开启整个启动流程    它会去加载META-INF/spring.factories  然后初始化初始化器 和 监听器    确定好 主类的位置
-  配置加载:  然后主要是为了构建环境   加载一些 配置文件  比如: application.properties/yml  如果有多个springboot配置文件  也可以通过spring.profile.active启用对应的配置文件
- 容器启动:  主要是通过IOC容器对象ApplicationContext  它进行初始化，运用之前加载的 `ApplicationContextInitializer` 还会对ApplicationContext进行预处理操作,还会把 `ApplicationListener` 注册到ApplicationContext当中 用于监听 启动的过程
- 自动装配:  这个阶段主要是通过 `@SpringBootApplication` 中包含的 `@EnableAutoConfiguration`，从 `META-INF/spring.factories` 里加载 `AutoConfiguration` 类 ， 还会依据 `@Conditional` 系列注解，筛选出符合当前环境的配置类  ， 最后将筛选后得到的配置类里的 Bean 定义注册到 Spring 容器中
- 组件初始化:   这个阶段主要用来做Bean 的创建与依赖注入
- 启动完成:  当容器初始化完成后,  它会发布 `ApplicationReadyEvent` 事件，表明应用已经可以对外提供服务了  ......

  >前面的备份和还原已经可以保证数据库数据安全了,但是还是需要人力去手动备份 但是如果忘记做了  出现安全隐患   所以除了手动备份还需要自动定期备份
  >
  >在sentos7系统中  一般采用cron job 来实现定期备份的功能    cron job是一个定时任务  类似于js的setTimeout  它负责完成定时任务的调度  用户可以根据需要自定义 时间  和间隔  它都会自动执行
  >

    ```

    ```

    ```
    比如:想设置每分钟备份一次
    */1 * * * * root mysqldump -uroot -proot --default-character-set=utf8 -B sc241201 > /usr/local/sc241201/sql/sc.$(date +\%Y-\%m-\%d-\%H:\%M:\%S).sql
    ```

  - 重启cron服务  让配置生效

### redis持久化方式

- RDB模式也叫半持久化模式,实际操作是redis服务端  每隔一段时间(可以控制) 生成一个子进程,  通过子进程负责将数据写入到来临时文件   等待写入成功后 再去替换之前保留的文件

  - 优点:
    - RDB模式是通过子进程进行持久化 不会影响redis服务端读写速度   执行效率是比较高的
    - RDB模式 做持久化后 只会保存一个RDB文件  对于后期数据恢复是比较容易的
  - 缺点:  容易出现数据丢失 是通过子进程每隔一段时间做持久化  万一这个时间间隔之内 redis宕机了   这段时间写入的数据就丢失了
- AOF模式 也叫全持久化模式, 底层是通过日志形式  去记录 redis服务端 执行的命令(增 删 改 查询不会记录)  如果后期redis宕机了  只需要重新执行一遍增删改的命令 即可

  - 优点:
    - AOF模式 具有较高的数据安全型  因为redis会有自己的同步策略(每秒同步 每修改同步...) 如果采用了每修改同步  那么每次修改都会记录日志  数据就不会出现丢失了
    - AOF模式 通过日志记录的  底层还会追加模式 每次记录写入的命令  不会损坏之前的存储的日志信息  
  - 缺点:  
    - AOF模式 每次写入 都会记录执行的命令  并发量很高时, 肯定会影响到redis服务端 执行速度   
    - AOF模式 保存时通过日志形式  肯定会存在很多个文件 比如每天一个文件    每个文件都需要恢复  数据量特别大时   恢复数据比较麻烦



### redis主从模式/主从复制

主从模式表示有一台主服务器(主要负责写入) 和 多台从服务器(只能读取 不能写入 具有主服务器备份)  这样 就算采用RDB模式  主服务器数据丢失了 也有很多从服务器数据备份

- 缺点:  
  - 如果从服务器宕机了  没有任何影响
  - 如果主服务器宕机了  数据虽然不会丢失  不能进行写入
  - 需要人工介入  将从服务器升为主服务器  才可以再次写入



### redis哨兵模式

哨兵模式   相当于主从模式的升级版  也有一台主服务器 和多台从服务器    只不过多了几个 放哨的(哨兵)   主要用于监控主从服务器可用性(底层会每隔一段时间向服务器发送一些数据包(心跳))  如果发现主服务器宕机了   这些哨兵就会投票决定（超过半数才会决定）   将哪台从服务器升为主服务器   好处在于不用人工介入了

- 缺点:
  - 只有一个主节点  也有一台服务器可以写入
  - redis并发量 和 容量无法做到扩容 



### redis集群模式

集群模式 它最大可以增加1000个节点(服务器)  这里面可以包括多台主节点 和多台从节点    这样redis整体性能(并发量 和容量 ) 会随着节点越多 也变得越高

- master: 主服务器(主节点)  负责写  可以是多个
- slave: 从服务器(从节点)  负责读   可以是多个   会保存主节点备份
- slot:  数据分槽(槽) redis一共有16384个槽 每个槽点管理一部分数据    这些数据分槽平均分配在所有的主节点上    这样主节点越多   每个主节点需要管理的 数据分槽就越少   在整体数据量不变的情况下    每个主节点  存储的数据 和 并发量也会越来越少



### redis缓存穿透  击穿 雪崩

- 缓存穿透: 在访问redis不存在的数据时  最后都只能访问关系型数据库比如:mysql  , 如果并发量特别高  这样mysql数据库的压力就很大了  
  - 解决方案:  可以在redis缓存中存储一些空对象  这些空对象一定设置过期时间   或者 可以使用布隆过滤器的数据结构
- 缓存击穿:  在同一个时间点  访问同一个数据时,这时redis的数据突然没了(可能被别人删除了 可能key失效了)  那么就会造成同一个时间 如果并发量特别高 就会同时访问mysql 造成击穿
  - 解决方案: 使用互斥锁确保只有一个请求去访问数据库 同时还要更新缓存
  - 解决方案: 也可以设置key永不过期
- 缓存雪崩:  在redis运行过程中  由于redis宕机了 或者redis所有key集体过期了  访问的所有操作 都全部集中到了mysql中 造成雪崩
  - 解决方案: 针对于宕机的问题  redis可以搭建主从  哨兵 或者集群模式     就算宕机了一台 还有很多台服务器
  - 解决方案: 针对于key都过期的问题    可以设置key不同的(随机处理)过期时间

    - 释放锁
  - 读取

### 什么是RabbitMQ

是一种应用程序和其他应用程序 通信的一种方式,通常来说消息发送者称之为生产者, 消息接收者 称之为消费者,  遵循先进先出(FIFO)原则    底层核心就是临时存储消息的队列     降低了 生成者和消费者之间耦合度    实现真正意义上的跨平台 跨系统的消息传递了    它核心主要包含: 生产者 消费者  交换机  队列   通过解耦的方式   实现生产者和消费者 进行异步通信



### rabbitMQ应用场景

- 项目和项目之间没有什么关系  但是有需要做数据传递  

  比如: 手机短信验证码(服务器可以生成验证码 通过第三方平台  发送至手机   MQ就通过生产者发送验证码到队列中  消费者就可以接收到  )

- 同时还可以把项目中 一些不需要立即返回的 而且 特别耗时的操作 通过MQ做异步处理  

  比如: 饿了么  美团  订单支付环节  需要等待用户付款结束  才可以走后续流程  这样其他的操作是不影响的   而且 用户支付的时间是无法控制的   这样就可以通过MQ来做处理 下单了到了支付了 向MQ发送一个支付消息   等到用户付款结束了  也会发送一个支付成功的消息   美团饿了么 监听到这个消息 就可以走后续 发货流程



### RabbitMQ消息模型

- 基本消息模型： 一个生产者发送消息到队列中,然后也只有一个消费者 去监听队列的消息 , 生产者和消费者是一对一的
- 工作队列：一个生产者 对应多个消费者   但是每个消息 只能被一个消费者消费      一般是会平均分配消息    后期可以设置能者劳   `channel.basicQos(1)`
- 发布订阅模式(fanout)： 一个生产者 发送消息  可以给多个消费者获取   还额外有一个组件 交换机    和 多个队列   每个队列 对应一个消费者   这样就可以保证 一个消息可以同时被多个消费者消费
- 路由模式(direct):  路由模式就是发布订阅模型升级版, 也具有多个消费者 一个交换机 多个队列 一个生产者, 区别在于 生产者将消息给交换机  要指定路由规则(路由key),消费者监听队列 要绑定交换机 必须提供正确的路由规则 才可以监听到消息
- 主题订阅模式(topic)：相当于路由模式升级版  核心组件都一样  区别在于  路由规则 可以使用通配符的方式 来匹配多个key  这样可以简化  绑定交换机多行代码
  - 通配符:  .*  只能匹配一个词    比如: java.*
  - 通配符:  .# 可以匹配一个或者多个词或者0个   比如: java.#



### RabbitMQ消息确认机制(ACK机制)

RabbitMQ有一个消息确认机制(ACK机制)   当消费者获取消息后 向rabbitMQ发送一个成功回执ACK   用于告知消费成功,  当消费者 获取消息后  由于各种原因没有使用   向rabbitMQ发送一个失败的回执 NACK  用于告知消费失败  则消息要重回队列  以便下一次再次获取...  

==注:前提想实现消息确认机制 必须要手动ACK==

### RabbitMQ死信队列(订单延时)

作用是用于 当消息发布到队列中 , 如果消息变成了`死信 `，它就会重新发布到一个新的交换机 这个交换机就是死信队列

- 什么情况消息会变成死信

  - 消息TTL过期：消息达到最大时才了 变成死信
  - 消息被拒绝:   比如设置接收消息时 是手动ACK   如果设置是否重回队列 时  设置有问题  如果没有重回队列  则变成死信

###  接口幂等性

就是表示针对于同一个接口地址 发次多少次重复请求，系统运行的状态 或者执行的结果 是保持一致的    如果结果保存不一致 则不是幂等性



### 为什么要实现幂等性

- 前端重复提交表单: 用于提交表单数据 由于网络波动 可能你认为服务器没有提交 所以会一致单击提交按钮  这时就会发送重复提交表单的问题
- 用户恶意进行刷单: 比如: 实现用户投票 ,如果用户针对于同一个人 进行重复投票   后端接口也会收到重复提交的信息 这种情况一定是非法的
- 接口超时重复提交:   一般是类似于调用第三方接口    为了防止 网络波动 超时 造成请求失败   都会添加重试机制  导致一次请求提交多次
- 消息进行重复消费: 当使用MQ的时候 , 如果MQ出现错误 导致没有及时提交消费信息   导致发生重复消费

**

### 实现幂等性方式

- 请求唯一标识:  为每个请求生成一个唯一的id (比如:UUID)  服务器处理请求时  先验证唯一的id是否一致的
- 数据库约束: 数据库唯一或主键约束  可以避免重复的数据
- 乐观锁:  更新数据时 检查版本号 (version)  匹配当前执行的版本是否一致  如果一样才能修改 
- http幂等方法:  表示Http的请求方式  get一定是幂等性    put和delete 有可能幂等的      post一定不是幂等的



###  什么是restFul

RestFul是一种软件设计风格，主要强调一个资源只能对应唯一的URL,  一个资源(可以是任意数据图片  任意数据 任意文件..) 无论实现新增 删除 修改 查询 只能使用一个URL, 最后通过请求方式不同 来区分具体要处理什么操作  是为了简化URL,   同时RestFul 几种不同请求方式  也可以用于实现幂等性

- get方式:  get方法用于获取资源, 无论发送多少次 对服务器数据不会更新 所以 一定是幂等的
- post方式:  post方法用于提交数据(新增)  每次执行请求都会新增数据   服务器的结果每次都不同  所以一定不是幂等的
- put方式:  put方法用于修改资源， 根据实际操作来确定是否满足幂等性
  - 幂等时:  当put请求 对目标资源修改时  多次调用修改重复的数据    最终结果时一致的  
  - 不幂等时:  当put请求对目标资源修改时   比如进行了累加操作（加薪） 多次调用导致结果肯定不一致的
- delete: delete方法用于删除资源， 根据实际操作来确定是否满足幂等性
  - 幂等时:  首次删除资源后   后续重复删除的时候 不会改变任何资源   结果保持一致
  - 不幂等时:  如果delete删除时    伴随着 级联操作 或者状态变更 (  删除订单  同时减少库存 ) 重复调用  库存可能每次都不同   所以结果也是不一致的

### 负载均衡

负载均衡是一种在计算机网络和系统架构中使用的技术  用于将请求均匀分布多个资源中, 使系统可以处理更多的并发请求, 如果其中一个资源出现故障  负载均衡也会自动将请求转发到其他可用的资源用

- 负载均衡 有哪些技术实现

- 负载均衡 可以使用什么算法
  - 轮询算法:  按照顺序逐一的分发请求   
  - 加权轮询:  根据服务器性能或者配置  添加权重  这样高性能的节点获取到更多请求
  - 哈希算法: 基于客户端的ip或者session ID 计算哈希值    保证相同的计算机 固定收到同一台服务器的资源  
  - 最少链接数:  优先选择获取链接数 最少的服务器 给你分配资源

### Sentinel 流控规则使用

原理就是通过监控应用流量的QPS或者并发线程数等指标，达到了指定的阈值时  就可以对流量进行控制  从而避免了流量高峰   保障了系统的高可用性   通常服务生产者需要配置流控

- QPS进行流控

- 并发线程数进行流控

  

- 流控模式:  主要表示流控的资源是一个还是多个

  - 直接: 默认值   访问的资源和流控的资源是同一个
  - 关联: 访问的资源 会关联其他资源   这样 不仅访问的资源 会被流控 关联的资源也会被流控
  - 链路:  类似于链表的头节点  微服务具有服务的调用链   只会针对于链路头 进行 流控 其他资源 不进行流控

- 流控效果:  是为了实现 如果 达到阈值 在什么时机显示流控结果

  - 快速失败: 默认值   只要达到阈值  立马进行流控
  - Warm up:  预热的意思  可以设置越热时间  让流量缓慢进入 
  - 排队等待:  可以设置一个时间  只要达到阈值了   让其等待一段时间再处理



###  Sentinel 熔断降级使用

sentinel除了流量控制以外,对服务调用之间 哪些不稳定的资源(访问时间很久, 出现异常,访问超时...) 可以进行熔断降级  也是保障高可用的重要措施 ， sentinel可以对不稳定的资源进行切断   防止服务出现整体雪崩  通常再服务的消费者配置

- 保护服务的手段:
  - 并发控制 来熔断

### 什么是JWT

Json Web Token (JWT): 它是跨域身份认值的解决方案，可以通过json对对象在前端和后端进行数据传递  ，主要用于实现用户登录授权, 一但用户登录成功，后端生成一个token(令牌)给前端 相当于给了前端一个授权码，以后前端后续发送的每个请求 都必须携带这个token, 而后端再执行方法之前  就会校验这个token合法(签名  有效期 非空)  如果验证通过才可以执行后续的代码  

- 流程

  - 前端输入用户得 账号信息  发送给后端

  - 后端 收到数据后 匹配数据库 验证账号信息 是否通过

  - 如果认证通过 生成 JWT token  通过json返回给前端

  - 前端进行本地保存 token

  - 以后前端发送得每一次请求 都要在请求头 上 携带 这个token

  - 后端添加了一个拦截器 会拦截发送得每次请求  拦截 之后验证请求头信息得token是否 正确

    - 如果拦截器验证得token有效 就可以执行后续逻辑
    - 如果验证得token 无效 就返回错误信息给前端 

  - 最后前端 还要添加 路由守卫功能 他可以控制前端哪些请求可以直接访问

    哪些请求需要登录后访问  如果没有登录 强制登录



### JWT组成

一个JWT实际上就是一个字符串,主要包含了三个部: Header(头信息),Payload(载荷),signature(签名), 中间通过 "." 进行连接  所以一个完成的jwt token  :  头信息.载荷.签名

- 头信息: 主要存放声明的令牌格式 和 加密算法

  ```
  {
      //alg 属性表示签名使用的算法
  	alg: "HS256",
  	//typ 属性表示令牌类型  JWT令牌统一写JWT
  	typ: "JWT"
  }
  ```

- 载荷(用户信息) : 主要包含一些默认字段 和需要传递给前端用户的数据  最后通过一个算法Base64  转换成字符串存储

  ```
  //默认字段
  iss : 签发人
  sub : 主题
  aud : 用户
  exp : 过期时间
  iat : 签发时间
  ...
  {
      sub: "登录认证",
      name: "易聪"，
      sex: "1",
      role: "普通用户"
  }
  ```

- 签名:  首先需要指定一个secret(密码) ,改密码保存再服务器中   这个通过头信息和载荷 通过base64进行加密后 通过.连接形成一个字符串    然后通过头信息里面声明的算法 进行加盐 组合加密   最后形成一个签名

  - (123 + 随机内容(盐值)) 进行加密 加密多少次

  



### SYN包攻击

SYN包攻击是一种网络攻击手段，攻击者向系统发送大量伪造的TCP连接请求(SYN包)，但不完成三次握手过程，导致系统耗尽资源无法处理正常的连接请求。

- 要避免SYN包攻击，可以考虑:
  - 配置网络设备和服务器，限制单个IP地址或IP地址范围对服务器的连接数，以防止过多的连接请求
  - 及时更新操作系统、应用程序和安全补丁，以修复已知的漏洞和弱点，提高系统的安全性
  - 使用防火墙和入侵检测/防御系统来监控和过滤网络流量，及时识别并丢弃可疑的SYN包

### 粘包问题

指的是发送方发送的多个数据包在接收方被合并成一个或多个不完整的数据包，导致接收方无法正确解析数据的现象

比如: 发送方连续发送的`A、B、C`三个数据包，接收方可能收到`AB、C`或`ABC`等合并后的结果



- 出现粘包的原因: 面向连接的 “流式协议”，它会将数据视为连续的字节流，而非独立的数据包   底层为了提高传输效率  会做一些优化

  - Nagle 算法: 合并小数据包，减少网络交互次数（默认开启）
  - 接收缓冲区:  接收方会先将数据存入缓冲区，若应用层未及时读取，后续数据会继续存入，导致多包数据 “粘” 在一起

- 解决粘包问题:   **给数据包定义明确的边界**，让接收方能够正确拆分数据

  - 长度前缀: 在每个数据包前添加一个固定长度的 “长度字段”（如 4 字节整数）
    - 发送方：先发送数据长度（如用 4 字节存储），再发送数据内容
    - 接收方：
      1. 先读取长度字段（如 4 字节），解析出数据的实际长度`L`。
      2. 再从缓冲区读取`L`字节的数据，作为完整数据包

  - 规定每个数据包的长度固定（如 1024 字节）
    - 发送方：不足固定长度时用填充符（如空格）补全
    - 接收方：每次从缓冲区读取固定长度的数据，去除填充符后解析
  - 分隔符法: 在每个数据包末尾添加一个特殊分隔符（如`\r\n`、`|`等，需确保分隔符不会出现在数据中）
    - 发送方：每个数据包后附加分隔符。
    - 接收方：从缓冲区中按分隔符拆分数据，直到读取到完整数据包。

  



### 七大软件设计原则

- 单一职责原则：
  一个类只负责一项职责，即一个类只有一个引起它变化的原因。这样做可以提高类的内聚性，降低耦合度，使得类更加可维护和可重用。

- 开放-封闭原则：
  软件实体（类、模块、函数等）应该对扩展开放，对修改关闭。这意味着可以通过扩展现有代码来满足新的需求，而不需要修改已有的代码，从而降低了系统的风险性和复杂性。

- 里氏替换原则：
  所有引用基类对象的地方必须能透明地使用其子类的对象。也就是说，子类对象可以替换父类对象并且程序行为不变。这样做有助于保持系统的一致性和可靠性。

- 依赖倒置原则：
  高层模块不应该依赖于低层模块，二者都应该依赖于抽象。抽象不应该依赖于具体实现细节，具体实现细节应该依赖于抽象。这样做可以降低模块之间的耦合度，提高系统的灵活性和可维护性。

- 接口隔离原则：
  客户端不应该被迫依赖它不使用的接口。一个类对另一个类的依赖应该建立在最小的接口上。这有助于减少不必要的依赖关系，使得系统更加灵活和易于维护。

- 合成/聚合复用原则：
  尽量使用对象组合，而不是继承来达到复用的目的。通过将现有类作为新类的成员对象，而不是通过继承现有类，可以更好地管理类与类之间的关系，降低耦合度。

- 迪米特法则：
  一个对象应该对其他对象有尽可能少的了解，不和陌生人说话。也就是说，一个对象应该对其他对象有限的了解，只和最亲近的朋友通信。这有助于降低类与类之间的依赖关系，提高系统的稳定性和可维护性。





### mysql使用limit性能会有问题吗

- 性能影响：当查询结果集非常大时，使用 LIMIT 可能会导致性能下降。因为 LIMIT 会在数据库中执行完查询后，再对结果集进行截取，这意味着数据库需要先执行完所有的查询操作，然后再返回所需的数据。如果数据量很大，可能会占用较多的内存和处理时间。

- 分页问题：LIMIT 主要用于分页查询，但是当数据发生变化（如删除、插入等），分页结果可能会出现问题。例如，如果在前一页数据中删除了某条记录，直接使用 LIMIT 进行分页查询可能会导致某些数据被漏掉或者重复显示。

- 解决方式: limit 有一个偏移量  避免使用大偏移量值   可以通过记录上一页的最后一个ID来进行分页查询
  合理使用索引
  通过explain查询sql语句的执行计划 来优化查询语句  不必要的排序操作，只查询需要的字段

### 什么是版本控制

再开发项目过程中, 项目文件会不断更新迭代 ，可能经常出现代码逻辑的问题  就需要回退到老版本 或者开发项目是多人协同开发的  每个人开发自己的模块   如果需要把这么多模块进行统一管理   就需要实现版本控制



### 实现版本控制的方式

- 集中化版本控制(SVN): 使用单一的服务器集中管理 ,保存所有配置 和文件 和修订版本  其他开发人员只需要安装对应的客户端  先去连接服务端   后面就可以正常更新和提交数据到服务端

  - 优点:  管理方便 只需要维护一台服务器 ， 还可以统一控制每个账号权限    操作简单 只需要 update和commit 
  - 缺点:  必须要连接到服务端 否则什么都做不了   如果项目比较重要 不允许回家办公  通常公司会使用svn通过局域网 搭建一个服务端   办公的时候 通过公司提供的电脑 连接服务端  去开发    ,还容易出现单点故障  
- 分布式版本控制(git): 相比之前svn  它git的客户端  其实都会保存 服务器所有代码  （相当于把服务器的完成代码镜像下载下来了）  这样 任何一个服务器出现问题 都可以利用任意一个客户端的代码  进行恢复

  - 优点:  每个客户端都具有完成的代码备份    不依赖于服务器    也可以支持离线操作
  - 缺点:   操作起来比较繁琐    权限控制难处理





### git常用命令

- init : 初始化仓库

- add：添加本地暂存区 

- commit：提交本地仓库

- push: 推送远程仓库

- pull ：如果远程远程具有本地仓库没有的代码  需要通过pull拉取最新内容到本地

- remote : 将本地仓库和远程仓库进行关联

- clone ：克隆项目到本地仓库

- 撤销命令

  - 如果提交本地仓库 没有推送远程仓库 

    - 通过reset撤销

  - 如果本地仓库和远程仓库都提交了

    ```sql
    -- 先查看提交记录，找到需要撤销的 commit ID（前几位即可）
    git log --oneline
    
    -- 再通过revert撤销指定提交id
    git revert <commit-id>
    
    -- 最后推送反向提交到远程，完成撤销
    git push origin <分支名>
    ```



### git如何解决版本冲突/git如何解决分支冲突

如果出现冲突了  会弹出一个窗口  要选择merge 进行不同版本代码合并(idea版本 本地仓库版本 远程仓库版本 )  要先和修改得代码开发人员  沟通好(统一好版本再提交   是提交你得 还是提交我得 )   如果找不到沟通人员   要不先不提交 要么先把代码保存一下   再拉取远程代码

### ERP系统/企业资源规划系统

ERP系统涉及的领域非常广泛，主要包括以下几个方面：
1.财务管理： 这是ERP系统的核心之一，涵盖了会计、成本控制、资金管理、预算管理等方面，帮助企业管理财务活动和资金流动。

2.供应链管理： ERP系统可以协调和优化供应链中的各个环节，包括采购管理、物流管理、库存管理等，从而提高供应链效率和降低成本。

3.生产管理： ERP系统可以管理生产过程中的各个环节，包括生产计划、物料需求计划、生产调度等，帮助企业实现生产过程的优化和控制。

4.销售与市场营销： ERP系统可以协助企业管理销售活动和市场营销活动，包括销售订单管理、客户关系管理、市场营销活动管理等，提升销售效率和客户满意度。

5.人力资源管理： ERP系统可以管理人力资源相关的各项工作，包括员工信息管理、薪酬管理、绩效管理、培训管理等，帮助企业合理配置人力资源并提升员工满意度。

6.项目管理： ERP系统可以协助企业进行项目规划、执行和监控，包括项目进度管理、资源分配、成本控制等，提高项目管理效率和成果。



### CRM系统/客户关系管理系统

CRM系统涉及的领域主要包括以下几个方面：

1.客户信息管理： CRM系统用于收集、存储和管理客户的基本信息，包括联系信息、购买历史、偏好等，以便企业更好地了解客户并提供个性化的服务。

2.销售管理： CRM系统可以帮助企业管理销售流程，包括销售机会跟进、销售活动管理、销售预测等，从而提高销售效率和业绩。

3.客户服务： CRM系统可以支持客户服务团队处理客户的问题、投诉和需求，包括客户服务工单管理、售后支持、客户反馈等，以提升客户满意度和忠诚度。

4.市场营销： CRM系统可以协助企业进行市场营销活动，包括客户分析、目标市场选择、营销活动执行等，帮助企业吸引新客户并保持现有客户的关注。

5.客户分析： CRM系统可以通过数据分析和报告功能帮助企业深入了解客户的行为和偏好，从而制定更有效的营销策略和客户管理策略。





### OA系统/办公自动化系统

OA系统（办公自动化系统）涉及的领域主要包括以下几个方面：
1.日常办公管理： OA系统可以帮助企业管理日常办公事务，包括邮件管理、日程安排、会议管理、文件共享等，提高办公效率和组织协作能力。

2.流程审批管理： OA系统可以支持企业内部各类审批流程的管理，包括请假申请、报销审批、合同签署等，实现审批流程的电子化和自动化，提高审批效率并减少错误。

3.文档管理： OA系统可以帮助企业管理和存储各类文档和资料，包括文档版本控制、权限管理、文档检索等功能，提高文档管理的效率和安全性。

4.项目管理： OA系统可以支持企业进行项目管理，包括项目进度跟踪、任务分配、团队协作等，帮助企业提高项目管理效率和成果。

5.人力资源管理： OA系统可以管理企业的人力资源相关事务，包括员工档案管理、招聘流程管理、绩效考核管理等，帮助企业合理管理人力资源并提升员工满意度。

6.知识管理： OA系统可以帮助企业管理和共享知识资产，包括知识库管理、培训资源管理、经验分享等，促进组织内部的知识交流和学习。
